# 🐧 Docker: Техническая реализация на уровне ядра Linux

> [!ABSTRACT] Философия изоляции
> Контейнер — это не виртуальная машина, а обычный процесс в Linux, на который наложены границы видимости (**Namespaces**) и лимиты потребления ресурсов (**cgroups**). Это обеспечивает «нативную» производительность без гипервизора.

---

## 1. Механизмы изоляции: Namespace

Пространства имен определяют, **что** процесс может видеть в системе. В ядре это реализовано через структуру `nsproxy` в дескрипторе процесса `task_struct`.



### Основные типы Namespaces
* **PID:** Изолирует ID процессов. Внутри контейнера ваше приложение — это `PID 1`.
* **NET:** Виртуализирует сетевой стек (интерфейсы, порты, маршруты).
* **MNT:** Позволяет процессу иметь собственную иерархию файловой системы (через `pivot_root`).
* **UTS:** Изолирует hostname и домен.
* **USER:** Позволяет процессу иметь `UID 0` (root) внутри контейнера, оставаясь непривилегированным пользователем на хосте.

> [!TIP] Инструмент (Arch Linux)
> Используй `lsns` для просмотра всех активных пространств имен и их связей с процессами.

---

## 2. Управление ресурсами: cgroups

Если Namespaces ограничивают видимость, то **cgroups** ограничивают **аппетиты** процесса. В Arch Linux используется **cgroup v2** (единая иерархия).



### Контроллеры ресурсов
* **CPU:** Использует планировщик **CFS**. Можно задать жесткую квоту (напр. 50% одного ядра) или относительный вес.
* **Memory:** Регулирует RAM. При достижении лимита `memory.max` вызывается **OOM Killer**, который завершает процесс.
* **I/O:** Ограничивает скорость чтения/записи (IOPS/BPS), предотвращая "забивание" диска одним контейнером.

---

## 3. Файловая система: OverlayFS

Docker использует слоистую структуру, чтобы экономить место и время запуска. Основа — драйвер **Overlay2**.



### Механизм Copy-on-Write (CoW)
* **Lowerdir:** Неизменяемые слои образа (ReadOnly).
* **Upperdir:** Слой записи конкретного контейнера.
* **Merged:** То, что видит процесс (объединение всех слоев).
**Суть CoW:** Если процесс хочет изменить файл из образа, ядро копирует его в `upperdir` и вносит правки там. Оригинал остается нетронутым.

---

## 4. Жизненный цикл и OCI

Docker следует стандартам **Open Container Initiative**.

1. **dockerd:** Принимает команды от пользователя.
2. **containerd:** Управляет жизненным циклом (pull образов, сеть).
3. **runC:** Низкоуровневая утилита, которая делает системные вызовы `clone()`, `unshare()` и `setns()` для создания контейнера.

---

## 5. Сетевой стек (Kernel Networking)

Изоляция сети достигается через манипуляции с интерфейсами и правилами фильтрации.

* **veth-пары:** Виртуальный "кабель", один конец которого в контейнере (`eth0`), а другой на хосте (`vethXXXX`).
* **docker0:** Программный мост (L2 Bridge), соединяющий все контейнеры.
* **NAT/iptables:** Правила на хосте, которые перенаправляют трафик с портов хоста (напр. 8080) на IP контейнера.

## 🛠 Investigation (Практика в терминале)

Чтобы увидеть эти механизмы своими глазами на Arch:
```bash
# Посмотреть в какой cgroup находится ваш процесс
cat /proc/self/cgroup

# Посмотреть veth-интерфейсы моста docker0
ip link show master docker0

# Проверить фильтры системных вызовов
cat /proc/self/status | grep Seccomp
```
