#  CPython Internals: Рантайм и Безопасность

> [!ABSTRACT] Системный взгляд
> Python — это не просто интерпретатор, а сложная стековая машина, работающая поверх C. Понимание того, как `PyObject` взаимодействует с памятью и GIL, позволяет оптимизировать высоконагруженные API и избегать утечек в контейнерах.

---

## 1.  Архитектура выполнения: Путь Байт-кода

Процесс трансформации кода в действие:
`Source` → `AST` (дерево) → `CFG` (графы переходов) → `Bytecode`.



### Визуализация стека PVM
В отличие от регистровых машин, **PVM (Python Virtual Machine)** — это стековая машина (LIFO). При вызове `jwt.decode()` состояние выглядит так:

| Стек значений (Value Stack) | Описание |
| :--- | :--- |
| `[Pointer to 'secret_key']` | TOP (загружен последним) |
| `[Pointer to 'token_string']` | Аргумент функции |
| `[PyFunctionObject: decode]` | Вызываемый объект |

> [!TIP] Исследование
> Используй модуль `dis`, чтобы увидеть инструкции перехода (напр. `JUMP_IF_FALSE_OR_POP`), которые управляют логикой твоей авторизации.

---

## 2.  Объектная модель: Все есть `PyObject`

Любая структура в CPython начинается с макроса `PyObject_HEAD`.

```c
struct _object {
    Py_ssize_t ob_refcnt;         // Счетчик ссылок
    struct _typeobject *ob_type;  // Указатель на тип
};
```

### Важные нюансы для QA/Dev:
* **Integer Interning:** Числа от **-5 до 256** кешируются. Сравнение `id is 257` вернет `False`, так как это разные объекты в памяти. **Всегда используй `==` для ID!**
* **PEP 393:** Строки в Python 3 имеют сложную иерархию для оптимизации памяти (Latin-1, UTF-16, UTF-32).

---

## 3.  Управление памятью и GC

CPython использует две стратегии:
1. **Reference Counting:** Немедленное удаление, когда `ob_refcnt == 0`.
2. **Cyclic GC:** Поиск циклических ссылок (напр. `User` ↔ `Session`).



### Поколения GC (Generations):
* **Gen 0:** Новые объекты. Сканируется чаще всего.
* **Gen 1:** Пережившие одну очистку.
* **Gen 2:** Долгоживущие объекты (напр. `app.config`).

> [!WARNING] Stop-the-world
> Сборка мусора в Gen 2 — дорогая операция. При обработке гигантских JSON-пайлоадов это может вызвать микро-фризы Flask-воркера.

---

## 4.  GIL и Параллелизм

**GIL (Global Interpreter Lock)** защищает `ob_refcnt` от состояний гонки.
* **CPU-bound (bcrypt):** Не ускорятся в потоках (Threads).
* **I/O-bound (запросы к БД):** Потоки эффективны, так как Python отпускает GIL во время ожидания сети.

**Решение для Flask/Gunicorn:**
Используй мультипроцессорность. Формула Corey Schafer:
`Workers = (2 * Cores) + 1`

---

## 5.  Иерархия памяти: PyMalloc

Для объектов **< 512 байт** (мелкие строки, словари) Python использует свой аллокатор, чтобы не беспокоить ядро лишний раз:
1. **Arenas (256 KB):** Выделяются через `malloc`.
2. **Pools (4 KB):** Внутри арен.
3. **Blocks:** Конкретные ячейки под данные.

##  Инструментарий для Arch Linux

Когда стандартные логи не помогают, используй системные утилиты:

* `gdb -p <pid>` — анализ C-стека зависшего воркера.
* `strace -p <pid> -e trace=network` — мониторинг сетевых вызовов.
* `ltrace -p <pid> -l libcrypto*` — отладка HMAC/RSA подписей в JWT на уровне системных библиотек.
