# Глубокий технический обзор Control Groups (cgroups) в Linux

> [!abstract] Определение и роль
> **Control Groups (cgroups)** — фундаментальный механизм ядра Linux для гранулярного управления, ограничения и изоляции ресурсов (CPU, память, I/O). 
> * **Namespaces** изолируют то, что процесс **видит**.
> * **Cgroups** ограничивают то, сколько процесс **потребляет**.

---

## 1. Теоретический фундамент и системная архитектура

Архитектура cgroups базируется на трех ключевых сущностях:

1.  **Иерархии (Hierarchies):** Древовидные структуры, к которым монтируются контроллеры.
2.  **Подсистемы/Контроллеры (Subsystems):** Модули ядра, отвечающие за конкретный ресурс.
3.  **Группы:** Узлы в иерархии, содержащие набор процессов и параметры лимитов.

### Механизм связывания: `css_set`
Для оптимизации ресурсов ядро использует структуру **`css_set`** (Control Group Subsystem Set). 
* Вместо хранения указателя в `task_struct`, используется хеширование уникальных комбинаций контроллеров.
* Это снижает накладные расходы при переключении контекста и поиске принадлежности процесса к группам.

---

## 2. Анализ ключевых подсистем (контроллеров)

Подсистемы интегрируются в **hot paths** ядра для мониторинга и ограничения ресурсов в реальном времени.

| Подсистема | Основные механизмы | Параметры (v1) |
| :--- | :--- | :--- |
| **CPU** | Планировщик CFS: доли (shares) и квоты (quotas). | `cpu.cfs_quota_us`, `cpu.cfs_period_us` |
| **Memory** | Управление страницами ОЗУ: Hard и Soft limits. | `memory.limit_in_bytes`, `memory.soft_limit_in_bytes` |
| **Cpuset** | Привязка (affinity) к ядрам CPU и узлам NUMA. | — |
| **Blkio** | Лимитирование пропускной способности (bps) и IOPS. | — |

> [!important] Различие лимитов памяти
> * **Hard limit:** Превышение при невозможности очистки вызывает **OOM Killer**.
> * **Soft limit:** Учитывается ядром только в условиях общего дефицита памяти в системе.

## 3. Практический интерфейс: Взаимодействие через VFS

Управление реализуется через виртуальную файловую систему, обычно монтируемую в `/sys/fs/cgroup`.

### Алгоритм ручного управления (на примере cgroups v1):

1.  **Создание группы** (ядро автоматически генерирует файлы параметров):
    ```bash
    mkdir /sys/fs/cgroup/memory/production_app
    ```
2.  **Установка лимитов** (запись в файлы параметров):
    ```bash
    echo 1073741824 > /sys/fs/cgroup/memory/production_app/memory.limit_in_bytes
    ```
3.  **Миграция процесса** (атомарное перемещение PID):
    ```bash
    echo 1234 > /sys/fs/cgroup/memory/production_app/tasks
    ```

## 4. Эволюция механизмов: Сравнительный анализ v1 и v2



| Характеристика | cgroups v1 | cgroups v2 (Unified) |
| :--- | :--- | :--- |
| **Топология** | Разные деревья для каждого контроллера. | Единая унифицированная иерархия. |
| **Размещение процессов** | В любом узле иерархии. | Только в "листовых" узлах (**No Internal Processes**). |
| **Именование** | Разрозненное (`memory.limit_in_bytes`). | Стандартизированное (`memory.max`). |
| **Управление** | Сложная синхронизация иерархий. | Атомарность операций и простая интеграция с `systemd`. |

> [!quote] Правило No Internal Processes
> В **v2** процессы могут находиться только в директориях, не имеющих дочерних групп. Это устраняет неоднозначность распределения ресурсов между родителем и потомками.

## 5. Заключение
Cgroups — это фундаментальное звено современной экосистемы Linux, превращающее систему в динамический набор изолированных контекстов и гарантирующее детерминированное поведение под высокой нагрузкой.
