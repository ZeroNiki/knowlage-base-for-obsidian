# 🧬 cgroups v2: Механика управления ресурсами Linux

> [!ABSTRACT] Суть технологии
> **cgroups (control groups)** — это инструмент ядра, который позволяет объединять процессы в иерархические группы и жестко ограничивать их доступ к CPU, RAM и I/O. Если Namespaces отвечают за "изоляцию видимости", то cgroups — за "гарантию ресурсов".

---

## 1. Архитектура: cgroup v1 vs. v2

В твоем **Arch Linux** по умолчанию используется **cgroup v2** (Unified Hierarchy). В отличие от v1, здесь нет разрозненных папок для каждого ресурса; всё дерево процессов управляется из единой структуры.

| Характеристика | cgroup v1 (Legacy) | cgroup v2 (Unified) |
| :--- | :--- | :--- |
| **Иерархия** | Раздельная для каждого контроллера | Единая унифицированная |
| **I/O** | Плохая поддержка Buffered I/O | Полный контроль (Writeback throttling) |
| **Модель управления** | Хаотичная | Принцип "Single Writer" (через systemd) |

---

## 2. Интерфейс: /sys/fs/cgroup

В Linux управление ресурсами реализовано через псевдо-файловую систему. Здесь нет бинарных команд, только чтение и запись в файлы.

* **Создание группы:** `mkdir /sys/fs/cgroup/my_app`
* **Добавление процесса:** запись PID в `cgroup.procs`.
* **Удаление:** `rmdir` (только если группа пуста).

---

## 3. Контроллеры (Лимиты в деталях)

### ⚡ CPU: Квоты и Приоритеты
1. **cpu.max:** Жесткий лимит (Hard Limit). Формат: `$quota $period`.
   * *Пример:* `50000 100000` — разрешить использовать 50мс из каждых 100мс (0.5 ядра).
2. **cpu.weight:** Относительный вес (1–10000). Определяет, кто получит больше ресурсов при 100% загрузке CPU.

### 🧠 Memory: Эшелонированная защита
1. **memory.max:** Абсолютный потолок. Превышение = немедленный **OOM Killer**.
2. **memory.high:** "Мягкий" лимит. При его достижении ядро начинает агрессивно забирать память у процесса (reclaim), замедляя его работу, но не убивая сразу.
3. **memory.low:** Гарантированный минимум. Ядро не имеет права забирать эту память для нужд других процессов.



### 💾 I/O: Контроль диска
В v2 наконец-то корректно ограничивается "отложенная запись" (Buffered Write). Вы можете задать лимиты в байтах (bps) или операциях (iops) через файл `io.max`.

---

## 4. Механизм управления: systemd

В Arch Linux основным менеджером cgroups является **systemd**. Он не любит, когда в `/sys/fs/cgroup` лезут руками.

**Правильный путь для Arch:**
Используй `systemd-run` для запуска процесса с ограничениями "на лету":
```bash
systemd-run --scope -p MemoryMax=500M -p CPUQuota=20% my_script.sh
```
Это создаст временный юнит, который будет виден в `systemd-cgtop`.

## 🧪 Практическая лаборатория (Obsidian Lab)

Попробуй ограничить свой текущий shell вручную (нужен root):

1. **Создаем группу:** `mkdir /sys/fs/cgroup/test_lab`
2. **Лимит 100MB:** `echo 104857600 > /sys/fs/cgroup/test_lab/memory.max`
3. **Входим в клетку:** `echo $$ > /sys/fs/cgroup/test_lab/cgroup.procs`
4. **Тест:** Попробуй запустить тяжелую компиляцию или браузер из этого терминала. Как только потребление достигнет 100MB, ядро пришлет `SIGKILL`.

## 📊 Инструменты диагностики

| Команда | Что показывает |
| :--- | :--- |
| `systemd-cgtop` | Нагрузка в разрезе контрольных групп (сервисов) |
| `cat /proc/[PID]/cgroup` | В какой именно группе находится конкретный процесс |
| `lsns` | Связь cgroups с пространствами имен |

> [!SUCCESS] Вывод для QA/DevOps
> Понимание cgroups позволяет тебе предсказывать поведение приложений. Если твой контейнер в Kubernetes падает с ошибкой `OOMKilled`, ты теперь знаешь, что это сработал лимит `memory.max` в файловой системе `/sys/fs/cgroup`.
