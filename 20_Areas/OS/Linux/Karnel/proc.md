# Интерфейс /proc: Архитектура и Конфигурация ядра

> [!abstract] Архитектурная роль
> **/proc (procfs)** — это виртуальная файловая система (VFS), существующая только в ОЗУ. Она служит стратегическим механизмом экспорта внутренних структур данных ядра в пространство пользователя, реализуя концепцию «всё есть файл» для интроспекции рантайм-состояний.

---

## 1. Природа и механизмы VFS

Данные в `/proc` не имеют физического представления на диске. Они генерируются ядром «на лету».

*   **Механизм `seq_file`**: Когда процесс вызывает `read()`, ядро перехватывает запрос и итерирует по внутренним структурам (например, `task_struct`), динамически создавая текстовый вывод.
*   **Когерентность**: Данные всегда отражают текущее состояние ядра на момент вызова.



---

## 2. Интроспекция процессов: `/proc/[pid]`

Фундаментальный инструмент для отладки без интрузивных отладчиков.

| Объект | Данные | Архитектурная значимость |
| :--- | :--- | :--- |
| **`status`** | UID/GID, VmRSS, сигналы | Агрегированное состояние и лимиты. |
| **`maps`** | Адресное пространство | Верификация **ASLR** и анализ библиотек. |
| **`fd/`** | Файловые дескрипторы | Поиск утечек ресурсов и анализ сокетов. |
| **`environ`** | Переменные окружения | Проверка runtime-конфигурации. |

> [!info] Углубленная работа с памятью
> Файл **`pagemap`** позволяет транслировать виртуальные адреса в Physical Frame Numbers (PFN), что критично для анализа производительности в **NUMA-системах**.

---

## 3. Системные метрики и Сетевой стек

Центральный хаб для сбора метрик производительности.

*   **/proc/meminfo**: Источник данных для `free`. Содержит параметры `Slab` (кэши ядра) и `Dirty` (память, ожидающая записи).
*   **/proc/stat**: Агрегированная активность CPU (user, system, iowait). Помогает выявить проблемы с оверкоммитом в виртуалках.
*   **/proc/net/**: «Источник истины» для сетевого стека. Содержит сырые таблицы маршрутов и TCP-соединений.

---

## 4. Интерфейс управления: `/proc/sys` (sysctl)

В отличие от остальной части `/proc`, эта поддиректория позволяет **изменять** параметры ядра без перезагрузки.

### Практический кейс: Управление ICMP
`echo 1 > /proc/sys/net/ipv4/icmp_echo_ignore_all` — система перестанет отвечать на ping.

> [!warning] Волатильность и Персистентность
> Изменения через `echo` в `/proc/sys` являются **временными**. Для сохранения настроек после перезагрузки необходимо использовать файл `/etc/sysctl.conf`.

---

## 5. Методология диагностики «Первого эшелона»

При возникновении аномалий экспертный подход требует изоляции уровней:

1.  **Исключение DNS**: Использование `ping -n` или `tracepath -n`. Задержки часто вызваны таймаутами резолвинга, а не потерей пакетов.
2.  **Верификация маршрутов**: Прямое чтение `/proc/net/route` для выявления дублирующихся шлюзов.
3.  **Проверка лимитов**: Чтение `/proc/[pid]/limits` для выявления причин падения высоконагруженных сервисов.

## Сравнительный анализ инструментов

| Инструмент | Источник в `/proc` | Роль |
| :--- | :--- | :--- |
| **cat / grep** | Почти всё | Прямая интроспекция сырых данных. |
| **ip / route** | `/proc/net/route` | Форматированное представление сети. |
| **top / ps** | `/proc/[pid]/stat` | Визуализация состояния процессов. |

---
**Заключительный вывод:** Понимание механизмов генерации данных через `seq_file` и волатильности `sysctl` — это разделительная черта между рядовым администратором и экспертом.
