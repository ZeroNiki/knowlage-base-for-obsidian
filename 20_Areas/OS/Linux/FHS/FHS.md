# ️ Архитектура данных в Linux: FHS и управление хранилищем

> [!abstract] Суть
> В основе Linux лежит предсказуемость, обеспечиваемая стандартом **Filesystem Hierarchy Standard (FHS)**. Это фундамент переносимости: благодаря ему администратор мгновенно находит конфиги на любом дистрибутиве.

---

## 1. Стандарт FHS: География системы

FHS разделяет данные по двум векторам: **Статические/Динамические** и **Разделяемые/Неразделяемые**.

| Путь | Назначение | Критическая роль |
| :--- | :--- | :--- |
| `/` | **Root** | Начальная точка VFS, основа всего. |
| `/bin` | Утилиты | Команды для всех (ls, cp), нужны для загрузки. |
| `/etc` | Конфигурации | **«Мозг» системы.** Статические настройки сервисов. |
| `/usr` | Вторичная иерархия | Данные пользователей и прикладное ПО. |
| `/var` | Динамические данные | **Логи**, БД, почта. Файлы, меняющие размер. |
| `/boot` | Загрузка | Ядро (`vmlinuz`) и конфиги GRUB. |
| `/dev` | Устройства | Интерфейсы к драйверам (например, `/dev/nvme0n1`). |



---

## 2. Концепция «Всё есть файл» (VFS)

Ядро использует слой **Virtual File System** как абстрактный интерфейс. Для приложения нет разницы между SSD, RAM или сокетом.

### Виртуальные интерфейсы ядра (Runtime):
* **`/proc`**: Окно в состояние процессов и параметров ядра.
* **`/sys`**: Иерархия управления оборудованием (PCI, USB, питание).
* **`/dev`**: Файловое представление «железа».

> [!tip] Pro-Tip
> Вы можете менять настройки ядра «на лету», просто записывая значения в файлы внутри `/proc/sys/`.

---

## 3. Inodes: Глубинная структура файлов

Файл в Linux — это не его имя, а **inode** (индексный дескриптор). Имя — лишь ссылка на номер inode.

* **Метаданные в Inode:** права (rwx), владелец, таймштампы, указатели на блоки данных.
* **Ловушка Inodes:** Если создать миллионы мелких файлов, место на диске (Гб) останется, но новые файлы создать не удастся — кончатся дескрипторы.

**Команда проверки:** `df -i` (показатель `IUse%` критичен для архитекторов).

## 4. Механизмы ссылок

| Критерий | Жесткие ссылки (Hard Links) | Символьные ссылки (Symlinks) |
| :--- | :--- | :--- |
| **Inode** | Тот же номер, что у оригинала. | Свой inode (содержит путь). |
| **Кросс-ФС** | Невозможно. | **Да** (между дисками/разделами). |
| **Директории** | Запрещены. | Разрешены. |
| **Удаление** | Данные живут, пока есть хоть 1 ссылка. | Ссылка «бьется» при удалении цели. |

## 5. Архитектура монтирования (`/etc/fstab`)



**Золотое правило:** Используй только **UUID** устройства. Имена `/dev/sda1` могут измениться при переключении дисков.

**Разбор строки fstab:**
`UUID=...  /data  ext4  defaults,noatime  0  2`
1. **Device**: UUID диска.
2. **Mount Point**: Куда подключаем.
3. **FS Type**: `ext4`, `xfs` (для нагрузок), `btrfs` (CoW/снапшоты).
4. **Options**: `noatime` ускоряет работу, отключая запись времени доступа.

> [!danger] Правило безопасности
> После правки fstab **всегда** делай `mount -a`. Если там ошибка, система не загрузится, а команда `mount -a` сообщит об этом сразу.

## 6. Troubleshooting: Руководство по выживанию

### Сценарий: "Диск переполнен"
1.  **Локализация:** `du -sh /var/log/* | sort -h`
2.  **Действие:** Архивация старых логов.
3.  **Выбор:** `xz -9` — максимальная компрессия (архитектурный выбор в пользу места, а не скорости CPU).

### Сценарий: "Файловая система стала Read-Only"
Это защитный механизм ядра при ошибках записи.
1.  **Лог ошибок:** `dmesg | tail`
2.  **Попытка пересборки:** `mount -o remount,rw /`
3.  **Проверка:** `fsck -y /dev/sdX`
