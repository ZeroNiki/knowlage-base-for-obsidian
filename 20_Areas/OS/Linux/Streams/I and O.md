# Универсальная модель I/O и механизмы контроля доступа в Linux

> [!abstract] Стратегический фундамент
> Концепция **«все есть файл»** позволяет унифицировать взаимодействие с дисками, оборудованием (ens33), пайпами и сокетами (ping/dig) через единый интерфейс байтовых потоков.

---

## 1. Универсальная модель ввода-вывода (I/O)

### Файловый дескриптор (FD)
**FD** — это неотрицательное целое число, индекс в таблице открытых файлов процесса. Он служит «зацепкой» для ядра при выполнении операций.



### Жизненный цикл данных: Системные вызовы
1.  **`open()`**: Инициализация. Ядро проверяет права и создает запись в системной таблице.
2.  **`read() / write()`**: Трансфер. Работа с «сырыми» байтами с минимальными накладными расходами.
3.  **`close()`**: Терминация. Освобождение FD (важно из-за лимитов на процесс).

### Стандартные потоки данных
| FD | Поток | Роль |
| :--- | :--- | :--- |
| **0** | `stdin` | Стандартный ввод (клавиатура или `<`). |
| **1** | `stdout` | Стандартный вывод (`>`). |
| **2** | `stderr` | Вывод ошибок (диагностика). |

> [!tip] Перенаправление (Redirection)
> Команда `2> error.log` подменяет FD 2 процесса на файл. Приложение продолжает писать в FD 2, не подозревая о смене цели.

## 2. Дискреционный контроль доступа (DAC)

Модель базируется на идентификаторах **UID/GID** (`/etc/passwd`).

### Классические разрешения (rwx)
Определяются для трех категорий: **User** (Владелец), **Group** (Группа), **Others** (Остальные).

### Специальные биты
*   **SUID (Set-User-ID):** Запуск с правами владельца файла (например, `/usr/bin/passwd` для записи в `/etc/shadow`).
*   **SGID (Set-Group-ID):** Наследование группы родительской директории новыми файлами.
*   **Sticky Bit:** Запрет на удаление чужих файлов в общей директории (например, `/tmp`).



### Математика umask
**Правило:** `Default_Mode & ~umask`.
Если `umask = 022`, то новая директория (`777`) получит права `755` (`rwxr-xr-x`).

## 3. Расширенные списки управления доступом (POSIX ACL)

ACL решают проблему гранулярности, когда rwx-триады недостаточно.

*   **Возможности:** Назначение прав конкретным UID/GID вне основных групп.
*   **Механизм Mask:** Устанавливает верхний предел (**ceiling**) эффективных прав. Если маска `r--`, то даже при наличии `rw-` в ACL, итоговым правом будет только чтение.

## 4. Атомарность и целостность данных

Атомарность — неделимость операции, исключающая «состояние гонки» (race conditions).

### Флаг `O_APPEND`
При записи логов несколькими процессами обычный `write()` рискует перезаписать данные. 
**`O_APPEND`** гарантирует на уровне ядра, что:
1. Перевод позиции в конец файла.
2. Сама запись.
...выполняются как **одна неделимая операция**.

> [!success] Рекомендация для High-load
> Используйте `O_APPEND` вместо `flock`. Это исключает риск «зависания» файла при падении процесса и работает быстрее за счет отсутствия лишних переключений контекста.

---

## Итоговый вывод
Синергия универсального I/O и многоуровневого контроля доступа обеспечивает гибкость и изоляцию. Файловые дескрипторы дают абстракцию, DAC/ACL — безопасность, а атомарность ядра — физическую сохранность данных при параллельной обработке.
