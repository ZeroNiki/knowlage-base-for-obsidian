# Механизмы сигналов в Linux: Асинхронное управление процессами

> [!abstract] Сигналы как абстракция
> В архитектуре Linux сигналы выступают в роли «программных прерываний» — высокоуровневой абстракции аппаратных исключений, позволяющей ядру изменять поток управления процесса.

---

## 1. Взгляд изнутри ядра: Жизненный цикл

Сигнал обрабатывается при переходе процесса из режима ядра в режим пользователя. Вся информация хранится в структуре `task_struct`.



### Фазы жизни сигнала
1.  **Generation (Генерация):** Инициируется ядром (сбой, таймер) или через `kill(2)`.
2.  **Pending (Ожидание):** Обновление битовой маски (**bitmask**) ожидающих сигналов в `task_struct`. Сигнал «висит», пока процесс не будет выбран планировщиком.
3.  **Delivery (Доставка):** Проверка маски перед возвратом в userspace. Передача управления обработчику.

### Типы сигналов
*   **Стандартные (1–31):** Не поддерживают очередь (только один бит в маске). Недетерминированы при высокой частоте.
*   **Реального времени (32–64):** Используют механизм очередей. Каждое вхождение фиксируется — критично для прецизионных систем.

## 2. Фундаментальные сигналы: Семантика и Безопасность

### Сравнительный анализ: SIGTERM vs SIGKILL

| Сигнал | № | Перехват | Реакция процесса (Graceful Shutdown) |
| :--- | :--- | :--- | :--- |
| **SIGTERM** | 15 | **Да** | Вызов деструкторов, закрытие сокетов, сброс буферов. |
| **SIGKILL** | 9 | **Нет** | Принудительное уничтожение ядром. Очистка невозможна. |

> [!danger] Практический нюанс: "So What?"
> Использование `kill -9` для сетевых демонов (например, DHCP-клиента) приводит к появлению **«зомби-аренд»** в файлах `dhcpd.leases`. Процесс не успевает освободить IP, что вызывает конфликты адресации.

*   **SIGHUP (1):** Традиционный сигнал для перечитывания конфигурации демоном.
*   **SIGQUIT (3):** Инициирует **core dump** — незаменимо для посмертной отладки сложных состояний.

## 3. Программная логика: `sigaction()`

Для надежного ПО устаревший `signal(2)` заменяется на `sigaction(2)`.

**Преимущества `sigaction`:**
*   **SA_RESTART:** Автоматический перезапуск прерванных системных вызовов.
*   **Маскирование:** Блокировка сигналов внутри самого обработчика (защита от рекурсии).

---

## 4. Критические нюансы: Async-Signal-Safety

Асинхронность — источник **deadlocks**. Внутри обработчика запрещено использовать нереентерабельные функции (`printf`, `malloc`).

> [!tip] Решение для Kernel Engineer
> Используйте системный вызов `signalfd(2)`. Он транслирует сигналы в события файлового дескриптора, позволяя обрабатывать их в стандартном цикле `epoll` вместе с сетевым трафиком.

### Проблема прерванных вызовов (EINTR)
Сетевые вызовы (`read`, DNS-запросы) могут быть прерваны сигналом, возвращая ошибку `EINTR`.
```c
// Правильная обработка в сетевом стеке
while ((n = read(fd, buf, size)) < 0) {
    if (errno == EINTR) continue; // Повтор при прерывании
    else break; // Реальная ошибка
}
```
*Игнорирование `EINTR` приводит к ложным диагнозам проблем со связью.*
