# 📊 Продвинутая архитектура Grafana: Observability-as-Code и оптимизация производительности

В современной SRE-практике визуализация данных — это не просто набор графиков, а критический компонент инфраструктуры, требующий инженерного подхода. Стратегический переход к парадигме **Observability-as-Code** диктует отказ от ручного манипулирования интерфейсом в пользу версионируемых и проверяемых артефактов. Визуализация должна проходить через те же циклы Code Review и CI/CD, что и основной код приложения, гарантируя воспроизводимость системы мониторинга в любой среде.

## 📐 1. Архитектура Grafana и подход Dashboards-as-Code

Использование UI для создания дашбордов в Production (**Click-ops**) является опасным антипаттерном. Этот подход лишен истории изменений, не позволяет проводить аудит через *Pull Requests* и делает невозможным быстрый откат к заведомо рабочему состоянию при инцидентах. В распределенных системах Click-ops неизбежно ведет к рассинхронизации между окружениями и деградации "источника истины".

### Механизм Grafana Provisioning

Для декларативного управления Grafana использует механизм Provisioning. При инициализации инстанс сканирует директорию `/etc/grafana/provisioning`, ожидая строго определенную структуру подкаталогов: `/datasources`, `/dashboards`, `/notifiers` и `/plugins`.

* **Data Sources:** YAML-конфигурации позволяют регистрировать источники (Prometheus, Loki) с предустановленными параметрами безопасности и таймаутами.
* **Dashboards:** Grafana отслеживает изменения в целевых папках (через параметр `checkInterval`). В Kubernetes-средах это часто реализуется через Sidecar-контейнеры (например, в составе *Grafana Operator*), которые монтируют ConfigMaps с JSON-моделями дашбордов в файловую систему Grafana.

### Инструменты генерации (Foundation SDK, Jsonnet, Terraform)

Современный стек генерации смещается от работы с сырым JSON к высокоуровневым абстракциям:

* **Jsonnet (Grafonnet):** Позволяет создавать шаблонизируемые библиотеки компонентов. Вместо копирования 500 строк JSON для каждой панели, инженер описывает функцию, что радикально упрощает поддержку сотен однотипных дашбордов.
* **Grafana Foundation SDK:** Актуальный инструмент, приходящий на смену Grizzly, позволяющий генерировать дашборды на Go, Python или TypeScript.
* **Terraform Provider:** Основной инструмент для управления инфраструктурными объектами (Folders, Permissions, API Keys) и разграничения доступа (RBAC).

## 🚀 2. Оптимизация производительности и борьба с тяжелыми PromQL запросами

Скорость отрисовки дашборда критична при разборе инцидентов. Неоптимизированные запросы за большие интервалы (30+ дней) не только создают задержки для инженера, но и могут вызвать **Out-of-Memory (OOM)** на бэкенде (Prometheus/Mimir) из-за попытки поднять в память миллионы семплов.

### Механизм переменных `$__interval` и `$__rate_interval`

Grafana динамически вычисляет шаг запроса (*step*), адаптируя его под разрешение экрана и выбранный диапазон времени:

* `$__interval:` Предотвращает избыточную детализацию. Если в панели 1000 пикселей, нет смысла запрашивать 100 000 точек; Grafana рассчитает оптимальный шаг, чтобы не перегружать браузер.
* `$__rate_interval:` Решает проблему "рваных графиков" (*gappy rates*). В отличие от фиксированных интервалов (например, `rate(metric[5m])`), эта переменная гарантирует, что в окно усреднения попадет как минимум два (а на практике — больше) семпла, учитывая текущий *step* и *scrape interval*.

> 💡 **Expert Tip:** Для стабильности запросов рекомендуется устанавливать интервал не менее 4x от `scrape_interval`. Это нивелирует риски пропуска единичных скрейпов и выравнивает шаг сетки.

### Query Splitting, Caching и Downsampling

При работе с долгосрочными данными (*Long-term storage*) необходимо использовать возможности бэкенда:

* **Query Splitting:** Разбиение тяжелого запроса на параллельные подзапросы за меньшие периоды времени.
* **Downsampling:** Использование предварительно агрегированных данных (доступно в Thanos и VictoriaMetrics). Вместо сырых данных за 30 дней с шагом в 15с, Grafana запрашивает 1-часовые или 5-минутные агрегаты, что ускоряет загрузку в десятки раз.

### Паттерны проектирования UI

Количество панелей прямо пропорционально нагрузке. Использование **Rows (строк)** с включенным параметром *"Repeat for variable"* позволяет реализовать **Lazy Loading**: панели в свернутых строках не инициируют запросы к БД, пока пользователь их не раскроет. Рекомендуемый лимит — не более 20 активных панелей в видимой области.

## 🚨 3. Динамический мониторинг: Переменные (Variables) и Алертинг

### Архитектура Variables и High Cardinality

Переменные на базе `label_values()` делают инструменты универсальными, но таят риски **High Cardinality** (высокой кардинальности). Попытка загрузить в выпадающий список тысячи `user_id` или `email` может парализовать интерфейс и нагрузить индекс Prometheus.

* **Решение:** Обязательная фильтрация на уровне запроса переменной. Вместо `label_values(instance)`, используйте `label_values(up{job="order-service"}, instance)`, чтобы ограничить выборку только актуальным контекстом.

### Grafana Alerting vs Prometheus Alerting Rules

Архитектурный выбор зависит от требований к отказоустойчивости:

* **Prometheus Alerting Rules:** Максимальная надежность. Вычисления происходят максимально близко к данным. Даже если Grafana недоступна, Alertmanager получит сигнал и отправит уведомление. Это стандарт для Fintech-систем и SLA по доступности транзакционного ядра.
* **Grafana-managed Alerts (Unified Alerting):** Гибкость и *Multi-source*. Позволяют объединять данные из разных источников (например, Prometheus + SQL) и использовать визуальный конструктор. Однако они зависят от работоспособности инстанса Grafana и сетевой связности с бэкендами.

## 🛠️ 4. Практика: Инфраструктурный код (IaC) руками

Приведенные ниже конфигурации являются эталонными для Production-развертывания с использованием принципа неизменяемости (*Immutability*).

### Datasource Provisioning YAML

Обратите внимание на `editable: false` — это блокирует случайные изменения через UI.

```yaml
apiVersion: 1
datasources:
  - name: Prometheus-Prod
    type: prometheus
    url: http://prometheus:9090
    access: proxy
    isDefault: true
    editable: false
    jsonData:
      httpMethod: POST
      queryTimeout: 45s
      timeInterval: "15s"
      # Кэширование на уровне проксирования запросов
      httpHeaderName1: "Cache-Control"
      httpHeaderValue1: "max-age=30"

```

### Dashboard JSON Template (фрагмент)

Фрагмент современной панели `timeseries` с использованием `$__rate_interval`.

```json
{
  "uid": "api-overview-prod",
  "title": "API Performance",
  "schemaVersion": 39,
  "refresh": "1m",
  "panels": [
    {
      "title": "Requests Rate (95th percentile)",
      "type": "timeseries",
      "targets": [
        {
          "expr": "histogram_quantile(0.95, sum by (le) (rate(http_request_duration_seconds_bucket{job=\"api\"}[$__rate_interval])))",
          "legendFormat": "p95 {{path}}"
        }
      ],
      "fieldConfig": {
        "defaults": { "unit": "s" }
      }
    }
  ]
}

```

## 🧠 Хардкорные вопросы по Grafana на собеседовании

* **Вопрос 1:** Дашборд за 30 дней загружается 5 минут. Ваши действия как архитектора?
* *Тезис ответа:* Проблема в объеме данных (*samples*). Необходимо: 1) Внедрить **Recording Rules** в Prometheus для тяжелых агрегаций. 2) Проверить наличие **Downsampling** в хранилище (Thanos/Mimir) и убедиться, что Grafana запрашивает агрегированные данные. 3) Использовать *Store Gateway Caching* для уменьшения задержек при обращении к Object Storage. 4) Проверить правильность вычисления `$__interval`.


* **Вопрос 2:** Как реализовать Multi-tenancy, если у нас 50 команд в одной Grafana?
* *Тезис ответа:* 1) Использование **Organizations** для полной изоляции. 2) Использование **Folders Permissions** в связке с группами в LDAP/AD через RBAC. 3) Настройка **Label-Based Access Control (LBAC)** на уровне Data Source (в Enterprise версии или через прокси), чтобы команда видела только свои метрики, даже если Prometheus общий.


* **Вопрос 3:** Панель показывает "Data Source Error" при нормальной работе БД. Куда смотреть?
* *Тезис ответа:* 1) Проверить логи Grafana на предмет **Context Cancellation**. Это происходит, когда запрос выполняется дольше, чем `queryTimeout` в настройках Data Source или глобальный таймаут браузера. 2) Проверить наличие лимитов на стороне прокси (Nginx/Ingress). 3) Убедиться, что количество серий в ответе не превышает `maxDataPoints`.
