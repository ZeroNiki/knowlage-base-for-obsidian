# Docker: CMD vs ENTRYPOINT

В Docker-контейнерах запуск основного процесса определяется двумя ключевыми инструкциями в `Dockerfile`. Их различие заключается в способе обработки аргументов командной строки и степени жесткости определения исполняемого файла.

### 1. Фундаментальные определения

* **ENTRYPOINT** (Точка входа): Определяет основную команду (исполняемый файл), которая будет запущена при старте контейнера. Она превращает контейнер в подобие исполняемого бинарного файла. Переопределить её сложнее (требуется флаг `--entrypoint`).
* **CMD** (Команда): Определяет аргументы по умолчанию, которые передаются в `ENTRYPOINT`. Если `ENTRYPOINT` не задан, `CMD` сама становится основной командой. Её легко переопределить, просто добавив аргументы в конце команды `docker run`.

### 2. Механика взаимодействия

Docker объединяет эти инструкции по следующей формуле:
`[ENTRYPOINT] + [CMD] = Итоговый процесс (PID 1)`

| Сценарий | ENTRYPOINT | CMD | Итоговая команда |
| --- | --- | --- | --- |
| **Только CMD** | *отсутствует* | `ping 8.8.8.8` | `ping 8.8.8.8` |
| **Только ENTRYPOINT** | `ping` | *отсутствует* | `ping` (ошибка, если нет аргументов) |
| **Обе инструкции** | `ping` | `8.8.8.8` | `ping 8.8.8.8` |

### 3. Переопределение при запуске

Главное различие проявляется в момент использования CLI (`docker run`):

1. **Переопределение CMD:**
Если запустить `docker run <image> google.com`, то `google.com` полностью заменит значение `CMD`, указанное в Dockerfile. Если был задан `ENTRYPOINT ["ping"]`, итоговая команда станет `ping google.com`.
2. **Переопределение ENTRYPOINT:**
Для смены исполняемого файла необходимо явно использовать флаг:
`docker run --entrypoint /bin/sh <image>`

### 4. Форматы записи (Exec vs Shell)

Существует две формы записи инструкций, которые критически влияют на обработку сигналов (например, `SIGTERM` при остановке):

* **Exec form (рекомендуется):** `["executable", "param1", "param2"]`
Процесс запускается напрямую как PID 1. Он корректно принимает системные сигналы.
* **Shell form:** `executable param1 param2`
Docker запускает процесс как подпроцесс оболочки: `/bin/sh -c "executable ..."`. В этом случае исполняемый файл не будет иметь PID 1 и не получит сигнал на корректное завершение от `docker stop`.

> [!TIP] Использование в DevOps
> Используйте **ENTRYPOINT** для задания неизменной логики контейнера (например, запуск сервера или утилиты), а **CMD** — для задания параметров по умолчанию, которые пользователь может захотеть изменить при тестировании или развертывании.
