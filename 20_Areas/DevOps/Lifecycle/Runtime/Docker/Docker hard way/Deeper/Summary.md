# Docker Cache: Механика инвалидизации слоев

Кэширование в Docker работает по принципу цепочки. Каждый слой жестко связан с предыдущим. Если в середине цепочки происходят изменения, вся последующая часть «отламывается» от кэша.

### 1. Ответ на задачу: Инвалидизация цепочки

**Вопрос:** Если изменить вторую строку на `RUN apt-get update -y && apt-get install -y curl`, возьмется ли третья строка (`RUN apt-get install -y nginx`) из кэша?

**Верный ответ:** **Нет, не возьмется.**

**Почему это происходит:**

1. Docker проверяет каждую инструкцию по порядку.
2. Как только он видит, что инструкция во второй строке изменилась, он помечает кэш для этой строки как невалидный и выполняет команду заново.
3. Поскольку третья строка теперь строится на базе **нового** (измененного) второго слоя, Docker больше не может гарантировать идентичность состояния файловой системы.
4. С этого момента Docker перестает даже проверять кэш для всех последующих строк и принудительно выполняет их одну за другой.

### 3. Резюме по оптимизации Dockerfile

Для эффективного использования кэша и ускорения сборки следует придерживаться следующих правил:

* **Расположение команд:** Размещайте инструкции, которые меняются редко (установка системных пакетов, настройка окружения), в начале `Dockerfile`. Команды, которые меняются часто (копирование исходного кода, запуск тестов), — в самом конце.
* **Минимизация слоев:** Объединяйте логически связанные команды в один слой через `&&`, чтобы уменьшить количество проверок кэша и итоговый размер образа.
* **Специфичность `COPY`:** Вместо `COPY . .` (которая сбрасывает кэш при любом изменении в проекте), копируйте сначала файлы зависимостей (например, `package.json` или `requirements.txt`), устанавливайте их, и только потом копируйте остальной код.
