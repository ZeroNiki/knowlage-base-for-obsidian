# Docker: Механика и оптимизация кэша

В Docker процесс сборки образа оптимизирован за счет использования кэшированных слоев. Каждая инструкция в `Dockerfile` формирует отдельный неизменяемый слой, который может быть переиспользован при последующих сборках.

### 1. Принцип работы кэширования

При запуске `docker build` движок Docker анализирует каждую строку `Dockerfile`:

1. **Сравнение инструкции:** Проверяется, совпадает ли текущая команда с той, что была выполнена ранее.
2. **Проверка контекста:** Для команд `COPY` и `ADD` Docker проверяет содержимое (хэш-суммы) файлов, копируемых в образ.
3. **Использование кэша:** Если изменений нет, Docker не выполняет команду заново, а мгновенно подключает готовый слой из локального хранилища (вывод в консоли: `Using cache`).

### 2. «Ловушка» кэша (Cache Busting)

Кэширование может привести к использованию устаревших данных, если внешние источники (репозитории пакетов) изменились, а `Dockerfile` — нет.

* **Проблема:** Если инструкция `RUN apt-get update` выделена в отдельную строку, она будет закэширована после первого выполнения. При добавлении нового пакета в следующей строке (например, `RUN apt-get install -y vim`), Docker возьмет результат `update` недельной давности.
* **Последствие:** Установка пакетов завершится ошибкой (404) или будут установлены старые версии с уязвимостями, так как списки пакетов в образе не обновились.

### 3. Best Practice: Объединение команд

Чтобы гарантировать актуальность данных и минимизировать количество слоев, применяется техника объединения связанных команд.

**Антипаттерн (плохо):**

```dockerfile
RUN apt-get update
RUN apt-get install -y nginx
```

**Рекомендуемый подход (хорошо):**

```dockerfile
RUN apt-get update && apt-get install -y \
    nginx \
    vim \
    && rm -rf /var/lib/apt/lists/*
```

> [!TIP] Оптимизация
> Объединение команд через `&&` гарантирует, что `install` всегда будет выполняться со свежим `update`. Добавление команды удаления кэша пакетов (`rm -rf ...`) в той же строке позволяет уменьшить итоговый размер слоя, так как временные файлы будут удалены до того, как слой станет неизменяемым.

### 4. Золотое правило «Домино»

Кэширование работает последовательно. Если кэш одной строки инвалидируется (из-за изменения команды или файлов), **все последующие строки автоматически теряют кэш** и будут выполнены принудительно.
