# User Namespaces и User Mapping: Безопасный суперпользователь

**User Namespaces** — это механизм изоляции, который позволяет переопределять идентификаторы пользователей (UID) и групп (GID). Это создает ситуацию, когда процесс обладает полными правами внутри своей среды, но остается абсолютно бесправным для остальной системы.

### 1. Концепция User Mapping (Отображение)

Суть технологии заключается в создании таблицы соответствия между «виртуальными» идентификаторами в контейнере и «реальными» идентификаторами на хосте.

* **Внутри контейнера:** Процесс видит себя как `UID 0` (root). Он может менять права доступа к файлам внутри контейнера, управлять системными службами и процессами.
* **На хост-системе (Arch Linux):** Ядро видит этот же процесс под обычным непривилегированным номером, например `UID 1001` (твой пользователь).

### 2. Почему это критически важно для безопасности?

В обычной конфигурации Docker (без User Namespaces), если процесс запущен от `root` в контейнере, он является `root` и на хосте. Если злоумышленник совершит «побег» (Breakout), он получит полный контроль над твоим компьютером.

**С включенным User Mapping:**

1. Злоумышленник взламывает приложение и получает `root` внутри контейнера.
2. Он находит способ прорваться через изоляцию на хост-систему.
3. **Итог:** Ядро хоста опознает его как обычного пользователя `nikita`. Он не сможет прочитать системные файлы, установить драйверы или удалить ядро, так как его «суперсила» была ограничена рамками контейнера.

### 3. Техническая реализация (The Hard Way)

Для реализации этого механизма Docker использует файлы конфигурации системы:

* `/etc/subuid`: Определяет диапазон доступных UID для отображения.
* `/etc/subgid`: Определяет диапазон доступных GID.

**Пример записи:**
`nikita:100000:65536`
Это означает, что для пользователя `nikita` выделено 65536 идентификаторов, начиная со 100000. Внутри контейнера `UID 0` превратится в `UID 100000` на хосте, `UID 1` в `100001` и так далее.

### 4. Резюме

User Namespaces решают главную дилемму безопасности: как дать приложению права, необходимые для работы, не подвергая риску всю инфраструктуру.

> [!CHECKPOINT] Итог изоляции
> * **Namespaces** — создают границы.
> * **Capabilities** — ограничивают действия.
> * **User Mapping** — подменяет личность, лишая «побег» смысла.
