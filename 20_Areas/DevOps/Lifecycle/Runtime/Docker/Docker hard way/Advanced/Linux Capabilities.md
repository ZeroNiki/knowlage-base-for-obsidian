В традиционной модели безопасности Linux существует бинарное разделение: либо всемогущий `root` (UID 0), либо ограниченный пользователь. **Capabilities** — это механизм ядра Linux, который разбивает огромную власть суперпользователя на отдельные, гранулярные привилегии.

### 1. Концепция гранулярных прав

Вместо того чтобы предоставлять процессу полный доступ ко всей системе, мы выделяем ему только те «суперсилы», которые необходимы для выполнения конкретной задачи.

* **`CAP_NET_ADMIN`**: Позволяет настраивать сетевые интерфейсы, таблицы маршрутизации и IP-фильтры.
* **`CAP_SYS_TIME`**: Позволяет изменять системное время.
* **`CAP_CHOWN`**: Позволяет менять владельца файлов (даже если процесс не является их владельцем).
* **`CAP_KILL`**: Позволяет отправлять сигналы завершения любому процессу в системе.

### 2. Управление в Docker

Docker по умолчанию запускает контейнеры с **белым списком** (whitelist) базовых привилегий, отсекая самые опасные.

* **`--cap-add`**: Расширяет возможности контейнера, добавляя конкретную привилегию.
* **`--cap-drop`**: Сужает возможности, отбирая даже стандартные права (принцип минимальных привилегий).
* **`--privileged`**: **Крайне опасно.** Флаг отключает всю систему фильтрации. Контейнер получает все права ядра и может получить доступ к устройствам хоста.

### 3. Практика: Эксперименты с привилегиями

Для закрепления материала проведи три эксперимента на своем Arch Linux.

#### Тест А: Попытка смены времени (Запрещено по умолчанию)

Обычный контейнер не может менять системные часы хоста.

```bash
docker run --rm alpine date -s "2030-01-01"
# Ожидаемый результат: date: can't set date: Operation not permitted

```

#### Тест Б: Выделение конкретной привилегии

Дадим контейнеру только право на работу со временем, не делая его полностью привилегированным.

```bash
docker run --rm --cap-add=SYS_TIME alpine date -s "2026-05-09"
# Результат: Команда выполнится (внимание: это может изменить время на хосте!)

```

#### Тест В: Максимальное ограничение (Hardcore Mode)

Запустим контейнер, отобрав у него абсолютно все привилегии, даже базовые.

```bash
docker run --rm --cap-drop=ALL alpine chown nobody /etc/hosts
# Результат: Даже базовые операции по смене владельца файла станут невозможны.

```

### 4. Резюме для DevOps

Использование `--privileged` в 99% случаев является признаком плохой архитектуры или лени. В профессиональной среде (DevOps) всегда следует использовать `--cap-add` только для тех функций, которые действительно требуются приложению (например, для инструментов мониторинга сети или глубокого тестирования производительности).

> [!CAUTION] Безопасность
> Предоставление лишних Capabilities — это прямой путь к **Container Breakout** (побегу из контейнера), когда злоумышленник получает контроль над всей хост-системой.
