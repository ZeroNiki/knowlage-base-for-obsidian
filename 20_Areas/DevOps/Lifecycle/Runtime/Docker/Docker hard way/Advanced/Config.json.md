# Низкоуровневая конфигурация: config.json и спецификация OCI

При работе с Docker на глубоком уровне (The Hard Way) важно понимать, что Docker-демон является лишь надстройкой над низкоуровневыми средами выполнения. Основным связующим звеном здесь выступает файл конфигурации в формате JSON.

### 1. Сущность config.json

Файл `config.json` — это манифест, содержащий полную спецификацию контейнера в соответствии со стандартами **OCI (Open Container Initiative)**. Этот файл считывается утилитой **runc** (дефолтная среда выполнения) для создания и запуска контейнера.

**Механика трансформации:**
Когда выполняется команда `docker run`, Docker-демон собирает все переданные аргументы и преобразует их в структурированный JSON-файл:

* Флаги `-v` превращаются в секцию `mounts`.
* Флаги `-p` и сетевые настройки уходят в конфигурацию `namespaces`.
* Флаги `--cap-add` и лимиты ресурсов (`-m`, `-c`) попадают в секцию `process` и `linux`.

### 2. Связь с механизмами переноса данных

Различие между методами сохранения данных в Docker напрямую связано с тем, как обрабатывается этот конфигурационный манифест.

> [[Save]]

#### Docker Export (Потеря метаданных) 

* **Действие:** Извлекает только файловую систему работающего или остановленного контейнера.
* **Результат:** Вы получаете архив (`.tar`) с файлами, но **теряете `config.json**`.
* **Последствие:** При последующем импорте через `docker import` вся информация о точках монтирования, портах, переменных окружения и командах запуска (`ENTRYPOINT`/`CMD`) будет утрачена.

#### Docker Save (Сохранение метаданных)

* **Действие:** Сохраняет образ целиком, включая все слои и метаданные.
* **Результат:** Внутри архива сохраняются файлы конфигурации для каждого слоя и общий манифест образа.
* **Последствие:** При загрузке через `docker load` образ восстанавливается в исходном виде со всеми заложенными инструкциями по запуску.

### 3. Значение для архитектуры (The Hard Way)

Понимание роли `config.json` позволяет:

1. **Отладка:** Видеть, какие именно параметры ядру Linux передает Docker-демон.
2. **Гибкость:** Запускать контейнеры напрямую через `runc` без использования Docker-демона, имея только корень файловой системы (rootfs) и правильно сформированный `config.json`.
3. **Безопасность:** Проверять финальный профиль безопасности (Capabilities, Seccomp, AppArmor), который будет применен к процессу.

> [!NOTE] Ключевое различие
> Если **Image** — это статический архив файлов + метаданные, то **config.json** — это «живой» план развертывания этих файлов в изолированной среде ядра.
