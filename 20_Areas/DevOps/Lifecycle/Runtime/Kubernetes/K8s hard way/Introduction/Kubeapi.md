# Компонент K8s: kube-apiserver

## 1. Общие сведения

`kube-apiserver` — это не просто фронтенд Control Plane, это центральный нервный узел кластера и единственная точка входа для всех операций.

С архитектурной точки зрения `kube-apiserver` является **stateless-компонентом**. Он не хранит состояние сам, делегируя эту задачу `[[etcd]]`. Именно эта «безгосударственность» позволяет масштабировать API-сервер горизонтально (High Availability), распределяя нагрузку между несколькими инстансами в высоконагруженных окружениях.

Любое действие — будь то вызов `kubectl` или работа внутреннего контроллера — технически является HTTP-запросом к этому бинарнику, который выступает в роли строгого привратника и валидатора схемы.

> [!IMPORTANT]
> **Архитектурная изоляция**
> В Kubernetes реализован жесткий запрет на прямое обращение любых компонентов к базе данных `[[etcd]]`. Только `kube-apiserver` имеет легитимное право записи и чтения из хранилища, что предотвращает коррупцию данных и нарушение логики безопасности.

### Аналитический слой «So What?»: Почему мы называем его «мозгами»?

API-сервер не просто передает данные; он обеспечивает соблюдение спецификаций и управляет Watch-механизмом. Контроллеры (например, Ingress или Gateway API) не опрашивают сервер постоянно, а «подписываются» на поток событий через механизм **WATCH**.

> [!FAILURE]
> **Последствия отказа**
> Если API-сервер недоступен, поток событий прерывается. Кластер «слепнет»: Ingress-контроллер не узнает о появлении новых подов, Gateway API не сможет обновить правила маршрутизации, а упавший под не будет перезапущен, так как `[[kube-scheduler]]` и `[[kube-controller-manager]]` потеряют связь с системой.

### Ключевые связи компонента

| Компонент | Тип взаимодействия | Зачем общаются |
| --- | --- | --- |
| `[[etcd]]` | Прямое чтение/запись | Единственный потребитель данных хранилища, синхронизация стейта. |
| `[[kube-scheduler]]` | Механизм WATCH | Получает задачи на планирование, отправляет обратно имя выбранной ноды. |
| `[[kube-controller-manager]]` | Механизм WATCH | Следит за изменениями ресурсов, поддерживает Desired State. |
| `[[kubelet]]` | Двунаправленный `[[mTLS]]` | Запрашивает спецификации подов (PodSpec), отчитывается о ресурсах ноды. |

## 2. Жизненный цикл запроса (Архитектура)

Для обеспечения безопасности в концепции «нулевого доверия» (Zero Trust) каждый запрос проходит через трехступенчатый конвейер. Пока запрос не прошел все стадии, он не существует для кластера и не попадает в базу данных.

```
[Входящий запрос] ──> [1. Authentication] ──> [2. Authorization (RBAC)] ──> [3. Admission Control] ──> [Запись в etcd]

```

### Детальный разбор этапов

1. **Authentication (Аутентификация):** Проверка личности (*кто пришел?*).
> [!NOTE] Архитектурный нюанс
> В Kubernetes отсутствует объект `User` в базе данных. API-сервер делегирует проверку личности внешним механизмам: клиентским сертификатам (X.509) или токенам (ServiceAccounts). Если личность не подтверждена, запрос отклоняется или обрабатывается как анонимный.


2. **Authorization (Авторизация):** Проверка полномочий (*что разрешено?*). Используется модель **RBAC**, где API-сервер сопоставляет субъекта с разрешенными «глаголами» (`verbs`: get, watch, create) и ресурсами. Здесь определяется область действия прав: `Role` ограничивает доступ конкретным namespace, а `ClusterRole` дает полномочия на уровне всего кластера.
3. **Admission Control (Контроллеры допуска):** Финальный фильтр, проверяющий запрос на соответствие политикам и бизнес-логике. Разделен на две фазы:
* **Mutating (Изменяющие):** Могут модифицировать запрос «на лету». Пример: `LimitRange`, который подставляет дефолтные лимиты ресурсов, если они не указаны в манифесте.
* **Validating (Валидирующие):** Проверяют запрос без изменений. Типичный пример — `ResourceQuota`. Если создание пода приведет к превышению квоты namespace по CPU или памяти, API-сервер прервет операцию и вернет ошибку `403 Forbidden`.

## 3. Безопасность и Сертификаты (mTLS)

В Kubernetes безопасность строится на взаимной проверке подлинности ([[MTLS]]). API-сервер не только проверяет клиентов, но и сам подтверждает свою личность. Без корректно настроенного PKI (Public Key Infrastructure) кластер не подниется.

> [!WARNING]
> Управление сертификатами — это задача «нулевого дня». Истечение срока любого из этих файлов полностью парализует управление кластером.

### Критические файлы для работы (в контексте kubeadm)

* **CA (Certificate Authority):** `ca.crt` / `ca.key` — корневой сертификат, фундамент доверия. Им подписываются все участники обмена.
* **Server TLS:** `apiserver.crt` / `apiserver.key` — серверный сертификат для защиты порта 6443. Обеспечивает шифрование трафика между API и внешним миром.
* **Etcd Client Certs:** `apiserver-etcd-client.crt` / `apiserver-etcd-client.key` — отдельные сертификаты для авторизации API-сервера перед `[[etcd]]`. Здесь API-сервер сам выступает в роли клиента.
* **Client CA File:** Определяет список доверенных CA, используемый сервером для проверки сертификатов, предъявляемых `[[kubelet]]`, `[[kube-proxy]]` и пользователями через `kubectl`.

## 4. Ключевые флаги запуска (Параметры бинарника)

Флаги определяют архитектурный облик кластера и его сетевые границы. При развертывании через `systemd` конфигурация выглядит следующим образом:

```ini
# Кусок файла /etc/systemd/system/kube-apiserver.service
ExecStart=/usr/local/bin/kube-apiserver \
  --etcd-servers=https://127.0.0.1:2379 \
  --tls-cert-file=/var/lib/kubernetes/apiserver.crt \
  --tls-private-key-file=/var/lib/kubernetes/apiserver.key \
  --client-ca-file=/var/lib/kubernetes/ca.crt \
  --service-cluster-ip-range=10.96.0.0/16

```

### Разбор параметров:

* `--etcd-servers` — указывает адреса хранилища. Для HA-кластеров здесь передается список endpoint-ов через запятую. Если связь со всеми нодами из списка будет потеряна, API-сервер перестанет изменять состояние кластера.
* `--tls-cert-file` и `--tls-private-key-file` — пути к файлам, обеспечивающим HTTPS-интерфейс. Без них невозможно организовать защищенный доступ к Control Plane.
* `--client-ca-file` — задает открытый ключ CA, которым должны быть подписаны сертификаты входящих клиентов (`[[kubelet]]`, `[[kube-scheduler]]`), чтобы сервер им доверял.
* `--service-cluster-ip-range` — определяет виртуальную сеть (CIDR) для объектов типа Service.
> [!TIP] Интеграция с сетью
> Этот флаг должен строго координироваться с настройками CoreDNS. Если диапазон указан неверно или пересекается с реальной сетью узлов, внутренний DNS кластера не сможет корректно резолвить имена сервисов в IP-адреса.
