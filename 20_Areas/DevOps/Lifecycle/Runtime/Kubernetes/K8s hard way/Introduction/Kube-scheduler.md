# Компонент K8s: kube-scheduler

## 1. Общие сведения

`kube-scheduler` — это стратегически важный «мозговой центр» Kubernetes, ответственный за интеллектуальное распределение ресурсов. Его роль выходит за рамки простого поиска свободного места; он обеспечивает стабильность кластера, предотвращая появление ресурсных «горячих точек» (hotspots) и гарантируя соблюдение инфраструктурных ограничений.

Этот компонент выполняет роль механизма декупликации: он полностью отделяет абстрактные требования приложения (CPU, RAM, Topology) от специфики конкретного оборудования, позволяя Control Plane оперировать логическими сущностями, а не физическими адресами.

Согласно архитектуре Control Plane, `kube-scheduler` является автономным бинарным файлом, работающим исключительно на мастер-нодах. Он функционирует по модели **Watcher**: через механизм Watch он отслеживает в [[Kubeapi]] появление подов, у которых поле `nodeName` остается пустым.

> [!IMPORTANT]
> **Границы ответственности компонента**
> Планировщик не взаимодействует с воркер-нодами напрямую и не видит те поды, которые были отклонены на этапе Admission (например, из-за превышения лимитов `ResourceQuota`). Если API-сервер пропустил под, задача планировщика — превратить «желаемое состояние» в «размещенную нагрузку».

## 2. Алгоритм планирования (Как выбирается Нода)

Процесс выбора узла — это детерминированный цикл, обеспечивающий переход пода из состояния `Pending` в статус запланированного. Этот цикл критически важен для минимизации рисков коррелированных отказов в распределенных системах.

```
[Под без nodeName] ──> [1. Фильтрация (Predicates)] ──> [2. Скоринг (Priorities)] ──> [3. Связывание (Binding)] ──> [Запись в API-server]

```

### Фильтрация (Predicates)

На этом этапе планировщик исключает узлы, не соответствующие требованиям пода, предотвращая размещение нагрузки на несовместимой инфраструктуре:

* **NodeSelector:** Базовая проверка соответствия меток (labels) узла.
* **Taints (Заразы) и Tolerations (Толерантности):** Механизм защиты узлов от размещения нежелательных подов. Существует три ключевых эффекта:
* `NoSchedule`: Новые поды без нужной толерантности не планируются.
* `PreferredNoSchedule`: Мягкий запрет (система постарается не занимать узел).
* `NoExecute`: Самый мощный эффект. Он не только блокирует планирование, но и немедленно выселяет (evict) уже запущенные поды, если у них нет соответствующей толерантности. Это ключевой инструмент при дренировании узлов.



### Скоринг (Priorities)

После фильтрации оставшиеся узлы ранжируются. Планировщик оценивает «качество» размещения, опираясь на:

* **Affinity и Anti-Affinity:** Используется для управления доменами отказа. Например, `podAntiAffinity` гарантирует распределение реплик по разным хостам (`kubernetes.io/hostname`) или зонам (`topology.kubernetes.io/zone`).
* **Pod Topology Spread Constraints:** Современный стандарт распределения. В отличие от Affinity, этот механизм использует параметр `MaxSkew` — максимально допустимую дельту (разницу) в количестве подов между топологическими доменами. Это позволяет добиться идеального баланса нагрузки.

### Связывание (Binding)

Завершающая фаза — «передача полномочий». Планировщик записывает выбранный узел в поле `nodeName` спецификации пода через `[[kube-apiserver]]`. В этот момент управление переходит от Control Plane к конкретному `[[kubelet]]` на выбранной ноде.

### Логика планирования: Инструментарий Архитектора

| Механизм | Фаза | Стратегическая цель |
| --- | --- | --- |
| **Taints / Tolerations** | Фильтрация | Изоляция инфраструктуры и защита Master-нод. |
| **NodeSelector** | Фильтрация | Жесткая привязка к специфическому оборудованию (GPU, SSD). |
| **Affinity / Anti-Affinity** | Скоринг | Управление близостью сервисов и отказоустойчивостью. |
| **Topology Spread Constraints** | Скоринг | Тонкая балансировка (`MaxSkew`) между зонами доступности. |
| **Binding (`nodeName`)** | Связывание | Фиксация решения и делегирование запуска агенту [[Kubelet]]. |

## 3. Безопасность и Сертификаты (mTLS)

В production-средах идентичность планировщика является целью высокого уровня для атак типа *Lateral Movement*. Поскольку `kube-scheduler` обладает правом записи в ресурс Pod (через процедуру Binding), компрометация его учетных данных позволяет атакующему произвольно размещать вредоносную нагрузку на любой узел кластера.

Безопасность обеспечивается через взаимную аутентификацию ([[MTLS]]). Для работы планировщику необходимы:

* **CA Certificates:** Для валидации подлинности API-сервера.
* **Client Certificates:** Индивидуальный клиентский сертификат (обычно с Common Name `system:kube-scheduler`), подписанный доверенным CA кластера.
* **Kubeconfig:** Файл (при развертывании через kubeadm расположен по пути `/etc/kubernetes/scheduler.conf`), объединяющий адреса, сертификаты и закрытые ключи.

> [!FAILURE]
> **Архитектурные риски**
> Ошибка в конфигурации или истечение сроков действия сертификатов (хранящихся в `/etc/kubernetes/pki`) приводит к немедленному параличу жизненного цикла приложений. Планировщик теряет возможность выполнять Watch-запросы, и кластер перестает реагировать на запросы к масштабированию, даже если ресурсы узлов полностью свободны.

## 4. Ключевые флаги запуска (Параметры бинарника)

Операционная настройка планировщика через флаги определяет идемпотентность и надежность Control Plane. При развертывании через Static Pods манифест располагается по пути `/etc/kubernetes/manifests/kube-scheduler.yaml`.

```yaml
# Пример структуры параметров в манифесте статического пода
spec:
  containers:
  - command:
    - kube-scheduler
    - --kubeconfig=/etc/kubernetes/scheduler.conf
    - --leader-elect=true
    - --authentication-kubeconfig=/etc/kubernetes/scheduler.conf
    - --authorization-kubeconfig=/etc/kubernetes/scheduler.conf

```

### Разбор параметров:

* `--kubeconfig` — путь к идентификационному файлу. Это «паспорт» планировщика, без которого невозможна любая авторизация в API-сервере.
* `--leader-elect` — критически важен для High-Availability (HA) сетапов. Флаг активирует механизм выбора лидера, гарантируя, что в многомастерной конфигурации активно только одно лицо, принимающее решения. Это обеспечивает идемпотентность состояния планирования и предотвращает Race Conditions при записи `nodeName`.
* `--authentication-kubeconfig` / `--authorization-kubeconfig` — позволяют планировщику делегировать проверку прав доступа (RBAC) обратно API-серверу. Это обеспечивает централизованный контроль безопасности, где внутренние компоненты подчиняются общим политикам доступа кластера.
