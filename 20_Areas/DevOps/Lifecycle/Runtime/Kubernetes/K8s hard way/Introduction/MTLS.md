# Технология безопасности: mTLS (Mutual TLS)

## 1. Общие сведения: Идентификация в мире эфемерности

В современных распределенных системах полагаться на сетевой периметр или IP-адреса — это архитектурное самоубийство. В динамическом кластере Kubernetes, где поды рождаются и умирают ежесекундно, IP-адрес является лишь временной меткой, а не идентификатором личности.

Концепция **Zero Trust** постулирует: сеть всегда скомпрометирована. Следовательно, каждая сущность должна доказывать свою подлинность криптографически. **mTLS (Mutual TLS)** — это стандарт де-факто для обеспечения безопасности «service-to-service». В отличие от классического TLS, где клиент проверяет только «паспорт» сервера, mTLS требует двусторонней аутентификации.

### Сравнение механизмов доверия

| Характеристика | Standard TLS (HTTPS) | Mutual TLS (mTLS) |
| --- | --- | --- |
| **Проверка сервера** | Да (Клиент проверяет Сервер) | Да (Взаимная проверка) |
| **Проверка клиента** | Нет (Требует L7 Auth: логин/токен) | Да (Криптографическая на L4) |
| **Основа доверия** | Публичные CA (DigiCert, LE) | Внутренний Private Root CA |
| **Идентификатор** | Доменное имя (FQDN) | Identity (ServiceAccount / SPIFFE) |

> [!IMPORTANT] **Identity-based Security**
> mTLS переносит фокус с вопроса «откуда пришел запрос?» (IP) на вопрос «кто этот запрос отправил?» (**Identity**). Это позволяет связать криптографическую подпись напрямую с субъектом в кластере через поле `Subject Alternative Name (SAN)` в сертификате.

## 2. Пошаговый алгоритм Handshake (Под капотом)

Процесс установления соединения в mTLS — это доказательство владения ключом (**Proof of Possession**) без раскрытия самого секрета.

1. **Client Hello & Server Hello:** Стороны обмениваются списком поддерживаемых Cipher Suites и случайными числами (`nonces`).
2. **Server Authentication:** Сервер отправляет свой сертификат. Клиент сверяет его цепочку с локально доверенным **Root CA**.
3. **Certificate Request:** **Критический этап.** Сервер посылает запрос: «Я доверяю списку этих CA, покажи свой сертификат».
4. **Client Authentication:** Клиент отправляет свой сертификат.
5. **CertificateVerify:** Клиент отправляет цифровой хеш всех предыдущих сообщений, подписанный своим **приватным ключом**. Сервер использует публичный ключ из сертификата клиента, чтобы проверить подпись.

> [!NOTE] **Ядро безопасности mTLS**
> Пакет `CertificateVerify` доказывает, что клиент не просто украл чужой публичный сертификат, а действительно владеет соответствующим ему приватным ключом.

6. **Session Key Generation:** После взаимной верификации генерируются симметричные ключи для шифрования основного потока данных.

## 3. mTLS в архитектуре Kubernetes

Kubernetes — это система, полностью управляемая сертификатами. Каждый запрос к [[Kubeapi]] проходит через механизм проверки личности.

### Роли API-сервера

* **API-сервер как Сервер:** Проверяет сертификаты `kubectl`, [[Kube-scheduler]] и [[Kubeproxy]]. Без валидного сертификата запрос отсекается на этапе Аутентификации (Who are you?), еще до проверки прав в **RBAC** (What can you do?).
* **API-сервер как Клиент:** При обращении к [[ETCD]] или [[Kubelet]] (например, для получения логов), API-сервер обязан предъявить свой сертификат. Это защищает узлы от несанкционированных команд со стороны скомпрометированных компонентов.

### Эволюция управления: Projected Volumes и Gateway API

Согласно материалам экспертов (в частности, Артура Крюкова), управление сертификатами в подах становится декларативным:

1. **Projected Volumes:** Вместо ручного монтирования статичных Secret-ов, K8s внедряет механизм инжекции сертификатов через **Projected Volumes**. Это позволяет системе автоматически ротировать сертификаты внутри контейнера, минимизируя риск утечки.
2. **Gateway API:** В отличие от старого Ingress, который терминирует TLS на границе, новый **Gateway API** позволяет использовать режим `Passthrough`. Это критично для mTLS, так как оконечное шифрование (End-to-End) сохраняется до самого приложения или Sidecar-прокси.

## 4. Преимущества, вызовы и «Ад ротации»

Внедрение mTLS — это сделка с операционной сложностью ради абсолютной безопасности.

### Преимущества

* **Иммунитет к MitM (Man-in-the-Middle):** Перехват трафика бесполезен без наличия приватных ключей обеих сторон.
* **Отсутствие статических секретов:** Сертификаты могут быть короткоживущими (часы или минуты), что делает их кражу бесполезной в долгосрочной перспективе.
* **Zero Trust на уровне L4:** Безопасность обеспечивается на транспортном уровне и не зависит от уязвимостей в коде приложения.

### Вызовы и риски

* **Management Hell:** Ротация сертификатов — главная точка отказа. Если сертификаты управления кластером (в `/etc/kubernetes/pki`) истекут, Control Plane «протухнет» и перестанет отвечать.
* **CPU Overhead:** Асимметричное шифрование при каждом новом Handshake создает заметную нагрузку на процессоры, что критично для систем с тысячами микросервисов.
* **Сложность отладки:** Ошибки уровня `bad certificate` или `unknown ca` требуют глубокого анализа цепочек доверия (`openssl`, `cfssl`), что на порядок сложнее анализа обычных логов HTTP.
