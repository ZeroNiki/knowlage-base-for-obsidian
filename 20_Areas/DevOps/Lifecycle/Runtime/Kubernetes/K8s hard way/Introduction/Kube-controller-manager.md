# Компонент K8s: kube-controller-manager

## 1. Общие сведения

В архитектуре Kubernetes `kube-controller-manager` (KCM) выполняет роль главного исполнительного органа Control Plane. Если [[Kubeapi]] — это пассивное хранилище состояний («почтовый ящик» или база данных), то KCM — это активный «мозг» и «исполнитель».

Любой YAML-манифест в кластере является лишь «желанием» (Desired State) пользователя. Само по себе описание ресурса в [[ETCD]] не приводит к его созданию в реальности. Именно `kube-controller-manager` непрерывно считывает эти конфигурации и предпринимает конкретные действия для их реализации.

* **Определение:** Это единый бинарный процесс, объединяющий в себе множество независимых контроллеров. С точки зрения архитектуры это компромисс: объединение упрощает развертывание и снижает нагрузку на систему, но создает единую точку отказа для всей логики управления кластером.
* **Механика Reconciliation Loop (Цикл сверки):** KCM работает по принципу непрерывного наблюдения. Он не просто циклически запрашивает данные, а «подписывается» на изменения через Watch API API-сервера.

```lucid
WHILE (cluster_is_running):
    1. WATCH: Подписка на события API Server (Events/Changes).
    2. OBSERVE: Считывание текущего состояния (Actual State) из etcd.
    3. DIFF: Сравнение Реальности с "Желанием" (Desired State) в манифесте.
    4. ACT: Выполнение вызовов к API для устранения расхождений.
END WHILE
```

> [!NOTE]
> Понимание KCM как механизма сверки позволяет перейти к детальному разбору конкретных «надсмотрщиков», каждый из которых отвечает за свой узкий участок логики в этом бесконечном цикле.

## 2. Основные встроенные контроллеры

Архитектурное решение объединить контроллеры в один бинарный файл продиктовано необходимостью минимизировать сетевой оверхед при общении с API-сервером. Каждый контроллер внутри KCM — это отдельный независимый поток (goroutine), реализующий специфическую бизнес-логику ресурсов.

### Ключевые внутренние контроллеры:

1. **Node Controller:** Компонент мониторинга инфраструктуры. Он анализирует состояние воркер-нод и, основываясь на статусах `Ready` / `NotReady`, накладывает соответствующие Taints. В случае окончательной потери связи инициирует процедуру Eviction (выселение) подов, сигнализируя планировщику[[Kube-scheduler]]  о необходимости переразмещения нагрузки.
2. **Replication / Deployment Controller:** Обеспечивает выживаемость приложений. Его задача — гарантировать, что количество реально запущенных инстансов строго соответствует параметру `replicas`. Если под терминируется, контроллер немедленно формирует запрос на создание нового через API.
3. **Endpoints / EndpointSlice Controller:** Магистраль сетевой связности. Он «прошивает» абстракцию Service реальными IP-адресами подов. Контроллер отслеживает изменения в селекторах подов и обновляет объекты Endpoints, чтобы трафик всегда попадал на актуальные и живые реплики ([[Kubeproxy]] использует эти данные).
4. **Namespace & ServiceAccount Controllers:** Фундамент безопасности и изоляции. `Namespace Controller` отвечает за каскадное удаление (cascading deletion) — когда при удалении пространства имен автоматически зачищаются абсолютно все связанные с ним ресурсы. `ServiceAccount Controller` автоматически создает дефолтные учетные записи и токены в каждом новом пространстве имен.

## 3. Безопасность и Сертификаты (mTLS)

Для работы в привилегированном режиме KCM требует обязательной аутентификации перед API-сервером. Все взаимодействие внутри Control Plane строго зашифровано и идентифицировано через `[[mTLS]]`.

* **Аутентификация (kubeconfig):** KCM использует файл конфигурации (обычно `controller-manager.conf`), содержащий его клиентские сертификаты. Без валидного kubeconfig контроллер не сможет авторизоваться и «подписаться» на события API-сервера.
* **Экосистема ServiceAccounts и Projected Volumes:** KCM играет центральную роль в безопасности рабочих нагрузок. Именно он использует флаг `--service-account-private-key-file` для подписания JWT-токенов.

> [!IMPORTANT]
> **Механизм защиты токенов**
> Сгенерированные JWT-токены являются основой механизма **Projected Volumes**. KCM подписывает токены с учетом полей аудитории (`aud`) и времени жизни (`expirationSeconds`). Это позволяет подам безопасно аутентифицироваться перед API-сервером или внешними сервисами, используя динамически монтируемые эфемерные тома, которые автоматически ротируются агентом [[Kubelet]].

## 4. Ключевые флаги запуска (Параметры бинарника)

Конфигурация KCM определяет не только функциональность, но и стабильность кластера в условиях высокой нагрузки или отказа узлов управления. При деплое через статические поды параметры объявляются в манифесте по пути `/etc/kubernetes/manifests/kube-controller-manager.yaml`.

### Таблица ключевых параметров запуска:

| Флаг | Назначение | Стратегический эффект |
| --- | --- | --- |
| `--kubeconfig` | Путь к сертификатам для авторизации. | Формирует доверенный канал управления (Chain of Trust) к API. |
| `--service-account-private-key-file` | Ключ для подписи JWT-токенов. | Обеспечивает работу Projected Volumes и криптографическую идентификацию подов. |
| `--root-ca-file` | Путь к сертификату CA кластера. | Позволяет контроллерам проверять подлинность [[Kubeapi]]. |
| `--leader-elect` | Активация механизма выбора лидера. | High Availability: предотвращает конфликты «split-brain», используя объект Lease в API. |
| `--use-service-account-credentials` | Использование уникальных SA для каждого контроллера. | Реализует принцип наименьших привилегий (Least Privilege) внутри потоков KCM. |

> [!TIP]
> **Механизм High Availability (HA)**
> Несмотря на запуск нескольких копий KCM на разных Master-нодах, активным в один момент времени может быть только один инстанс. Используя распределенные блокировки (`Leases`), контроллеры соревнуются за право владения «замком» в API-сервере. Если текущий лидер теряет связь, один из Standby-экземпляров мгновенно перехватывает управление, обеспечивая непрерывность цикла сверки.
