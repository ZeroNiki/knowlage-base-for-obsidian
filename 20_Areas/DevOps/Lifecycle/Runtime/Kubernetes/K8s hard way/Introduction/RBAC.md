## Аналитический обзор: RBAC в Kubernetes

RBAC (Role-Based Access Control) — это система управления доступом, которая отвечает на три вопроса: Кто, Что делает и С чем.

### 1. Карта сущностей и логика взаимодействия

Безопасность кластера строится на триаде: **Subject → Verb → Resource**.

- **Субъекты (Subjects):**
    
    - **User:** Внешняя сущность, определяемая через CN (Common Name) сертификата. K8s не хранит базу пользователей, а доверяет внешним CA.
        
    - **ServiceAccount:** Внутренняя сущность для приложений, использующая токены.
        
- **Действия и Ресурсы:** Операции (`get`, `list`, `watch`, `create`, `update`) над объектами API (`Pods`, `Secrets`, `Ingress`).
    
- **Связующее звено:** `Role` (что можно) и `RoleBinding` (кому можно).
    

> **Инсайт уровня эксперта:** Золотой стандарт безопасности — использование **ServiceAccount Token Projection**. Это временные токены с ограниченным сроком жизни (`expirationSeconds`) и привязкой к потребителю (`audience`), что делает их кражу бесполезной в долгосрочной перспективе.

### 2. Сравнение областей применения (Scope)

Изоляция — главный метод ограничения «радиуса поражения» (blast radius).

|**Уровень**|**Инструменты**|**Назначение**|
|---|---|---|
|**Namespace**|`Role`, `RoleBinding`|Изоляция команд и сред (например, `work` или `sample`).|
|**Cluster**|`ClusterRole`, `ClusterRoleBinding`|Инфраструктурные компоненты (Ingress, мониторинг), требующие доступа ко всему кластеру.|

**Гибридный паттерн:** Использование `ClusterRole` совместно с `RoleBinding`. Это позволяет стандартизировать права на уровне кластера, но ограничивать их действие конкретным неймспейсом для пользователя.

### 3. Критические векторы атак и эскалация привилегий

Некоторые права являются «точками невозврата» для безопасности:

1. **Доступ к Secrets:** Прямой путь к правам администратора, так как там хранятся ключи TLS и пароли БД.
    
2. **Создание Pods:** Злоумышленник может запустить под, смонтировать в него `ServiceAccount` более привилегированного сервиса и украсть его полномочия.
    
3. **Манипуляция трафиком:** Право на `NodePort` (порты 30000+) позволяет выставить внутренние сервисы наружу, минуя периметр.
    
4. **Ресурсный DoS:** Создание бесконечного числа ресурсов без квот парализует кластер.
    

### 4. Практические паттерны защиты

Принцип **Least Privilege** (наименьших привилегий) реализуется через:

- **Разделение ответственности (Separation of Concerns):** На примере Gateway API, где права разделены между провайдером инфраструктуры (шлюзы), оператором (порты/TLS) и разработчиком (маршруты трафика).
    
- **Принудительные ограничения (Guardrails):** Использование `ResourceQuota`. Если квота активна, API-сервер не создаст под без указания `limits` и `requests`, что повышает стабильность системы.
    
- **Изоляция контроллеров:** Настройка контроллеров так, чтобы они видели только необходимые им секреты (механизм `watchNamespace`).
    

### 5. Слепые зоны RBAC

Важно помнить, что RBAC — это не полная защита, а лишь один из слоев:

- **Отзыв прав (Revocation):** K8s не умеет нативно отзывать сертификаты пользователей. Решение — использование **OIDC**, где сессия аннулируется на стороне Identity Provider.
    
- **API vs Data:** RBAC защищает API-сервер, но не мешает уже запущенному поду сканировать внутреннюю сеть кластера.
    
- **Отсутствие гранулярности:** Нельзя разрешить видеть только определенные поля внутри одного объекта (например, только один ключ в Secret).
    

### Задачи для самопроверки (Checklist)

**1.Аудит (ReadOnly):**Ограничение прав инженера.

Создайте `Role` в неймспейсе `work` только с глаголами `get`, `list`, `watch` для ресурсов `pods/log`. Это позволит проводить диагностику без риска удаления данных.

**2.Изоляция контроллера:**Защита от компрометации.

Создайте `ClusterRole` для Ingress, но привяжите её через `RoleBinding` только к конкретному неймспейсу. Это ограничит доступ к сертификатам в случае взлома контроллера.

**3.Блокировка периметра:**Сетевая безопасность.

Настройте `ResourceQuota` с параметром `services.nodeports: "0"`. Это физически запретит открывать внутренние порты во внешнюю сеть.

> **Итоговый вывод:** Для построения стратегии **Defense in Depth** (эшелонированная оборона) дополняйте RBAC контроллерами допуска (OPA Gatekeeper/Kyverno) и сетевыми политиками (`Network Policies`).
