# ️ Архитектура и низкоуровневые механизмы мониторинга: Prometheus, Grafana, Alertmanager

В проектировании отказоустойчивых систем с интенсивностью в миллионы активных временных рядов (*time series*) глубокое понимание внутренних механизмов сбора и хранения данных — это базовое требование для выживания инфраструктуры.

---

## ️ 1. Архитектура TSDB (Time Series Database) в Prometheus

Локальное хранилище Prometheus оптимизировано для обеспечения консистентности записи, однако в высоконагруженных окружениях локальный диск служит лишь буфером перед передачей данных в системы долгосрочного хранения (LTS): **VictoriaMetrics**, **Thanos** или **Grafana Mimir**.

### Внутренняя структура хранения и механизмы Block Directory

Данные организуются в блоки (`Block Directory`), которые по умолчанию охватывают **2-часовые интервалы**.

```
data/
├── 01BKGV7JBM9BMH9MM6565BG9A1/   <- Уплотненный блок (2h+)
│   ├── chunks/                   <- Сжатые сэмплы (метрики)
│   │   └── 000001
│   ├── index                     <- Индекс лейблов (Inverted Index)
│   └── meta.json                 <- Метаданные блока (время, версия)
└── wal/                          <- Write-Ahead Log (Текущие данные в RAM)
    ├── 00000001
    └── checkpoint.000000

```

Каждый блок содержит сжатые сэмплы в файлах `chunks`, инвертированный индекс лейблов в `index` и метаданные. Ключевым процессом является **Compaction (уплотнение)**: со временем короткие 2-часовые блоки фоновыми потоками объединяются в более крупные (до 2 недель), что оптимизирует использование диска и ускоряет работу индексов.

### Write-Ahead Log (WAL) и восстановление

Механизм WAL фиксирует все входящие данные в хронологическом порядке до их обработки в RAM. Архитектурная деталь: файл WAL удаляется только после того, как текущий блок данных в памяти успешно «нарезан» (*cut*) и зафиксирован на диске. Это гарантирует отсутствие потери данных при внезапном сбое питания или падении процесса (`SIGKILL`).

### Механизмы сжатия (Gorilla Algorithm)

Использование алгоритмов сжатия позволяет Prometheus достигать среднего показателя в **1.37 байта на сэмпл**:

* **Delta-of-delta compression** используется для временных меток: вместо полных значений (*timestamp*) сохраняются лишь отклонения в интервалах опроса. Если интервал стабилен, дельта равна нулю, что занимает минимум бит.
* **XOR compression** применяется для значений с плавающей точкой (`float64`), минимизируя избыточность при хранении схожих или последовательно меняющихся метрик за счет операции исключающего ИЛИ с предыдущим значением.

### Проблема High Cardinality

В финтех-секторе использование динамических лейблов вроде `user_id` или `order_id` — это прямой путь к **«кардинальной бомбе»**. Каждая уникальная комбинация меток порождает физически новый временной ряд. Рост индекса ведет к лавинообразному потреблению RAM и неизбежному OOM-клину.

> ️ **Архитектурный стандарт:** Использовать только ограниченные наборы (*bounded sets*) для меток. Идентификаторы сессий или пользователей должны агрегироваться до уровня приложения или выноситься в системы распределенного трейсинга и логирования (Jaeger, Loki).

##  2. Модели сбора и Язык запросов (PromQL под капотом)

Prometheus придерживается **Pull-модели**, что позволяет серверу самостоятельно контролировать нагрузку на инфраструктуру и защищаться через `Scrape Limits`.

### Фундаментальные типы метрик

* **Counter:** Монотонный счетчик (только инкремент, сбрасывается в 0 при перезапуске приложения).
* **Gauge:** Текущее значение в данный момент времени (может как расти, так и падать).
* **Histogram:** Группировка событий по заданным корзинам (*buckets*) на стороне клиента.
* **Summary:** Расчет квантилей на стороне клиента (не подлежит агрегации).

### Histogram vs Summary (Deep Dive)

Главное отличие — в возможности агрегации. `Summary` рассчитывается *client-side*, и его квантили (например, p95) нельзя статистически корректно усреднять по нескольким инстансам приложения.

`Histogram` работает иначе: это набор обычных счетчиков-корзин с техническим лейблом `le` (*less-than-or-equal*). Для расчета 95-го процентиля в распределенной системе необходимо использовать функцию `histogram_quantile()` поверх агрегированных через `sum()` корзин.

```promql
# Архитектурно правильный расчет p95 latency для распределенного сервиса
histogram_quantile(
  0.95, 
  sum by (le) (
    rate(http_request_duration_seconds_bucket{job="api-gateway"}[$__rate_interval])
  )
)

```

>  **Важно:** Для расчета квантилей в PromQL необходимо использовать функцию `rate()`, а не `increase()`. Математика `histogram_quantile` опирается на линейную интерполяцию относительных пропорций корзин в единицу времени, а не на абсолютные значения приращения.

### Lookback Delta и Staleness

PromQL использует **5-minute Lookback Delta**. Если за последние 5 минут для ряда не было найдено ни одного сэмпла (и не был получен явный маркер устаревания), он считается «исчезнувшим» (*stale*) из графиков. Это фундаментальное правило объясняет, почему метрики «дребезжат» или прерываются при редких или нестабильных интервалах опроса.

##  3. Node Exporter и Системный уровень Linux

Сбор системных данных должен осуществляться с минимальным накладным расходом (*overhead*), чтобы не вносить искажения в работу бизнес-приложений.

### Механика сбора через procfs и sysfs

Node Exporter спроектирован так, чтобы полностью избегать тяжелых вызовов внешних команд через `exec` (таких как `df`, `free`, `top`). Вместо этого библиотека `procfs` на Go напрямую считывает псевдофайловые системы ядра Linux:

* Данные о памяти: парсинг `/proc/meminfo`
* Статистика I/O дисков: парсинг `/proc/diskstats`
* Метрики процессора: парсинг `/proc/stat`

Это переводит операцию в плоскость легковесного чтения из памяти ядра, радикально снижая нагрузку на CPU.

### Textfile Collector

Для метрик, которые не являются потоковыми (например, статус аппаратного RAID-массива, версии прошивок, время завершения бэкапа или результат работы cron-задач), используется `Textfile Collector`. Он сканирует директорию с `.prom` файлами и отдает их содержимое при общем скрейпе:

```bash
# Пример генерации кастомной метрики через cron-скрипт
echo "backup_status{type=\"db\"} 1" > /var/lib/node_exporter/textfile_collector/backup.prom

```

##  4. Alertmanager & Grafana: Маршрутизация и Dashboards as Code

### Alertmanager: Глушение шторма и логика меток

Алертинг строится на трех архитектурных концепциях:

1. **Grouping (`group_by`):** Схлопывание сотен однотипных событий в одно уведомление (например, падение 50 подов одного микросервиса группируется по лейблу `alertname` и `service`).
2. **Group Wait:** Технологическая пауза (например, 30с) для сбора связанных алертов перед отправкой, чтобы инженеру пришло одно структурированное сообщение вместо лавины уведомлений.
3. **Inhibition (`inhibit_rules`):** Подавление зависимых уведомлений. Например, если пришел алерт `DatacenterDown`, срабатывает правило, запрещающее отправку алертов уровня `InstanceDown` или `ServiceDegraded` для узлов в этом датацентре.

>  **Критическая ошибка:** Ошибка в PromQL-запросе алерта, агрегирующая метрики без сохранения ключевых лейблов (например, использование `sum()` вместо `sum by (cluster, service)`), лишает Alertmanager возможности правильно маршрутизировать, группировать или подавлять уведомления.

### Grafana и переменная `$__interval`

Использование `$__interval` в запросах критично для производительности интерфейса. Эта переменная динамически подстраивает шаг (`step`) в зависимости от временного диапазона и разрешения экрана. Это предотвращает падение вкладки браузера по памяти (не пытаясь отрисовать 100 000 физических точек на 1000 пикселей экрана) и снижает нагрузку на TSDB.

## ️ 5. Практика: Инспекция и Дебаг руками в CLI

Умение проверить экспортер вручную позволяет локализовать проблему между «приложение не отдает метрику» и «Prometheus не может ее забрать».

### Прямой опрос экспортера через curl

```bash
# Проверка доступности Node Exporter и фильтрация конкретной метрики
curl -s http://127.0.0.1:9100/metrics | grep node_cpu_seconds_total

```

### Автоматизация Provisioning Data Source в Grafana

Пример конфигурации (`/etc/grafana/provisioning/datasources/prometheus.yaml`), блокирующей редактирование источника данных через интерфейс (`editable: false`) для соблюдения GitOps-подхода:

```yaml
apiVersion: 1
datasources:
  - name: Prometheus-Fintech
    type: prometheus
    url: http://prometheus-server.monitoring.svc.cluster.local:9090
    access: proxy
    isDefault: true
    editable: false
    jsonData:
      httpMethod: POST
      queryTimeout: 30s
      timeInterval: "15s"

```

## ️ Финтех-чеклист по Observability для собеседования

| Задача / Проблема | Инженерное решение | Архитектурный смысл |
| --- | --- | --- |
| **Расчет Error Budget** | `rate(errors_total[30d]) / rate(requests_total[30d])` | Определение процента отказов относительно SLA на основе Counter-метрики. |
| **Угроза OOM на Prometheus (64GB RAM)** | Внедрение `metric_relabel_configs` с действием `drop` | Отсечение высококардинальных динамических лейблов *до* момента их сохранения в TSDB + лимит `sample_limit` на таргет. |
| **Мониторинг Serverless / Ephemeral Jobs (< 2 сек)** | Интеграция **Pushgateway** | Короткоживущие процессы сами «толкают» метрики в буфер, откуда Prometheus их забирает по стандартной Pull-модели. |
| **Проблема Staleness (Зависание графиков)** | Запись **Staleness Markers** | При падении таргета Prometheus генерирует специальный маркер устаревания. График обрывается мгновенно, не дожидаясь 5-минутного таймаута Lookback Delta. |
