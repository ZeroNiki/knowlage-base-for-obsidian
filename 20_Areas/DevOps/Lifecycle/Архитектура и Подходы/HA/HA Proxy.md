# HAProxy (High Availability Proxy)

**HAProxy** — это сверхбыстрый, надежный и депизированный балансировщик нагрузки (Load Balancer) и прокси-сервер для протоколов TCP (L4) и HTTP/HTTPS (L7). Это стандарт де-факто в DevOps для распределения трафика между бэкендами.

> [!abstract] Почему именно он?
> HAProxy использует событийно-ориентированную (event-driven) однопоточную архитектуру. Из-за отсутствия контекстных переключений между потоками он способен переваривать сотни тысяч запросов в секунду (RPS), потребляя минимум оперативной памяти.

---

## 1. Балансировка на уровнях L4 vs L7

HAProxy умеет работать в двух режимах, и это критически важно понимать для конфигурации:

### Режим TCP (Layer 4)

Балансировщик работает на транспортном уровне. Он не лезет внутрь пакета и не знает, какой там протокол (HTTP, SSH, база данных или Redis).

* **Как работает:** Просто перенаправляет TCP-пакеты от клиента к серверу на основе IP и порта.
* **Плюс:** Максимальная скорость и производительность.
* **Минус:** Нельзя настроить роутинг по путям (например, `/api` на один сервер, а `/static` на другой) или прочитать куки.

### Режим HTTP (Layer 7)

Балансировщик работает на уровне приложения. Он полностью парсит HTTP-запрос (заголовки, куки, URI, тело запроса).

* **Как работает:** Читает запрос, может терминировать SSL (расшифровывать HTTPS на себе), проверять авторизацию или перекидывать трафик в зависимости от URL.
* **Плюс:** Умный роутинг и гибкость.
* **Минус:** Требует больше ресурсов процессора, так как парсинг текста — тяжелая операция.

## 2. Основные алгоритмы балансировки

Как именно HAProxy выбирает, на какой сервер отправить текущий запрос:

* `roundrobin` (По умолчанию): Серверы получают запросы по очереди. Если у серверов разные мощности, можно задать им вес (`weight`), тогда мощный сервер получит больше запросов.
* `leastconn`: Запрос улетает на тот сервер, у которого прямо сейчас меньше всего активных TCP-соединений. Идеально для тяжелых и долгих запросов (например, скачивание файлов или сложные SQL-отчеты).
* `source`: Хеширует IP-адрес клиента. Это гарантирует, что пользователь с конкретным IP всегда будет попадать на один и тот же сервер бэкенда (важно, если сессии хранятся локально на сервере, а не в Redis).

## 3. Анатомия конфигурационного файла (`haproxy.cfg`)

Конфиг HAProxy логически делится на 4 основные секции. Вот шаблон базовой структуры:

```haproxy
global
    # Настройки самого процесса (логи, пользователи, лимиты)
    maxconn 4096
    user haproxy
    group haproxy

defaults
    # Таймауты и параметры по умолчанию для всех остальных секций
    log     global
    mode    http
    timeout connect 5s
    timeout client  50s
    timeout server  50s

frontend my_front
    # Точка входа для клиентов. На каком порту и IP слушаем трафик
    bind *:80
    # Правило (ACL): если путь начинается с /api, отправляем на backend_api
    acl is_api path_beg /api
    use_backend backend_api if is_api
    # Все остальное отправляем на дефолтный бэкенд
    default_backend backend_web

backend backend_web
    # Список серверов, куда перенаправлять трафик
    balance roundrobin
    # check включает health-checks (проверку доступности)
    server web1 192.168.1.10:8080 check
    server web2 192.168.1.11:8080 check

backend backend_api
    balance leastconn
    server api1 192.168.1.20:9000 check
    server api2 192.168.1.21:9000 check

```

## 4. Что такое Health Checks в HAProxy?

HAProxy не просто вслепую шлет трафик. Параметр `check` в секции `backend` заставляет его регулярно пинговать целевые серверы.

> [!success] Механизм самолечения (Failover)
> Если один из серверов бэкенда падает или начинает отдавать ошибки, HAProxy мгновенно исключает его из балансировки. Пользователи этого даже не заметят — их запросы молча перенаправятся на оставшиеся живые серверы. Как только сервер оживает, HAProxy плавно возвращает его в строй.
