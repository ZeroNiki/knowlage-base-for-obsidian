# Kubernetes High Availability (HA) Cluster

**Kubernetes HA Cluster** — это конфигурация, при которой управляющий контур кластера (**Control Plane**) задублирован на нескольких серверах (минимум 3 мастер-ноды). Это гарантирует, что падение одной или даже двух мастер-нод не приведет к остановке деплоя и управления контейнерами.

---

## 1. Как ведут себя компоненты Control Plane в HA?

Нельзя просто запустить три копии мастеров и надеяться, что всё заработает само. Разные компоненты K8s балансируются совершенно по-разному:

### API Server (`kube-apiserver`) — Active-Active

* **Состояние:** *Stateless* (не хранит состояние внутри себя, всё пишет в etcd).
* **Как работает в HA:** Абсолютно независим. Можно запускать сколько угодно копий на разных мастерах.
* **Балансировка:** Перед ними **обязательно ставится внешний балансировщик трафика** (например, тот же **HAProxy** + **Keepalived**). Все воркер-ноды (`kubelet`) и админы через `kubectl` общаются только с IP-адресом балансировщика, а тот раскидывает запросы по живым API-серверам.

### Scheduler и Controller Manager — Active-Passive

* **Состояние:** *Stateful логика* (если два шедулера одновременно попытаются назначить один и тот же под на разные ноды, начнется хаос).
* **Как работает в HA:** Они запускаются на всех мастерах, но активен в один момент времени **только один узел**.
* **Балансировка:** Используется механизм **Leader Election (выборы лидера)**. Компоненты внутри себя создают специальный объект `Lease` (аренду) в etcd. Тот, кто успел её захватить, становится главным (`leader`) и работает. Остальные спят и проверяют etcd каждые несколько секунд. Если лидер упал, другой узел перехватывает аренду и становится активным.

## 2. Сердце HA кластера: etcd и [[Кворум]]

**etcd** — это распределенное key-value хранилище, где лежит *вообще всё* состояние твоего кластера (конфиги, секреты, поды, сервисы). Нет etcd — нет Kubernetes.

Поскольку etcd использует алгоритм консенсуса **Raft**, для него железно действует правило нечетного кворума, которое мы разбирали ранее.

> [!math] Формула кворума для etcd
> 
> $$Q = \lfloor N/2 \rfloor + 1$$
> 
> 
> * Кластер etcd из **3 нод** выдерживает падение **1 ноды** ($3 - 2 = 1$).
> * Кластер etcd из **5 нод** выдерживает падение **2 нод** ($5 - 3 = 2$).
> 
> 
> *Именно поэтому минимальное количество мастеров для полноценного Kubernetes HA — строго **3 ноды**.*

## 3. Два типа топологии HA кластера

При развертывании кластера (например, через `kubeadm`) нужно выбрать, где именно будет жить etcd:

### Вариант 1: Stacked etcd (Совмещенная топология)

Компоненты Control Plane и базы etcd запущены **на одних и тех же** трех серверах. Каждая мастер-нода содержит полный стек.

* **Плюсы:** Легко настраивать, нужно меньше серверов (минимум 3 штуки для HA). По умолчанию разворачивается именно так.
* **Минусы:** Сбой по питанию одной ноды уносит с собой и API-сервер, и узел etcd. Риск потери кворума выше, если железные ноды нестабильны.

### Вариант 2: External etcd (Выделенная топология)

Управляющий контур разделен: у тебя есть 3 сервера под мастера (там крутятся API, Scheduler, Controller Manager) и отдельно 3 (или 5) изолированных серверов **только под etcd cluster**.

* **Плюсы:** Максимальная изоляция и отказоустойчивость. Падение мастера вообще никак не трогает базу данных.
* **Минусы:** Сложная настройка, нужно администрировать в два раза больше инфраструктурных серверов и настраивать гору TLS-сертификатов между ними.

## 4. Чек-лист: Что нужно для создания K8s HA

> [!check] Инфраструктурный минимум для HA:
> 1. [ ] **3 Master Nodes** (для обеспечения нечетного кворума etcd).
> 2. [ ] **Worker Nodes** (минимум 2-3 для распределения нагрузок приложений).
> 3. [ ] **Выделенный балансировщик (Load Balancer)** — например, пара серверов с HAProxy + Keepalived, которые дают единый виртуальный IP (VIP) для доступа к `kube-apiserver`. Без этого воркеры не смогут переключаться между мастерами.
