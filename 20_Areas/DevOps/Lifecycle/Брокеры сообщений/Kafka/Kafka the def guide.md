# 🪐 Apache Kafka: Архитектурный Deep Dive для системных инженеров

Данная заметка деконструирует внутренние механизмы Apache Kafka с позиции системного программирования и эксплуатации высоконагруженных распределенных систем. Фокус на структурах данных, системных вызовах и тюнинге ядра ОС.

---

## 1. Дисковая подсистема: Append-Only Log и Структура файлов

Выбор модели хранения данных в Kafka продиктован механикой работы современных дисковых накопителей. В то время как традиционные СУБД используют B-деревья (B-Trees) для обеспечения произвольного доступа, Kafka полагается на **Append-Only Log** (журнал записи в конец).

* **Sequential vs Random I/O:** Использование B-деревьев требует сложности $O(\log N)$ и неизбежно влечет за собой произвольное позиционирование головок HDD (Random Disk Seeks). В противоположность этому, последовательная запись (Sequential Write) в Kafka имеет сложность $O(1)$ и сопоставима по скорости с пропускной способностью сетевого интерфейса. Отказ от поиска (seek) на диске — это фундамент, позволяющий одному брокеру обрабатывать терабайты данных без деградации производительности.

### Анатомия директории партиции

Каждая партиция представлена набором сегментов в файловой системе:

* `.log`: Бинарный сегмент данных (по умолчанию 1 ГБ). При малых скоростях записи сегмент может оставаться открытым дольше периода ретеншена, что важно учитывать при планировании дисковой емкости.
* `.index`: Разреженный индекс (sparse index). Маппинг `offset -> position`. Вместо индексации каждого сообщения, Kafka создает запись в индексе через каждые `log.index.interval.bytes` (дефолт: 4096 байт). Это позволяет удерживать индекс в RAM и быстро находить физическую позицию в логе.
* `.timeindex`: Используется для поиска по метке времени (timestamp), что необходимо для работы политик удаления и семантического поиска.

### Механика Log Compaction

При `Cleanup Policy: Compact` Kafka сохраняет только последнее состояние для каждого ключа. Очистка (Cleaning) происходит в фоне через процесс *Log Cleaner*, который мерджит сегменты, удаляя устаревшие записи с одинаковыми ключами. Фактическое удаление (Purge) происходит после истечения `delete.retention.ms`.

> **Связующее звено:** Высокая скорость записи была бы бесполезна без возможности эффективной передачи данных из Page Cache в сетевой стек без участия CPU.

## ⚡ 2. Оптимизация на уровне ОС: Page Cache и Zero-Copy

Kafka спроектирована так, чтобы минимизировать накладные расходы JVM на управление памятью и Garbage Collection (GC), делегируя кэширование ядру ОС (VFS layer).

### Page Cache и системный тюнинг

Использование системного кэша страниц вместо JVM Heap позволяет избежать пауз GC при работе с огромными объемами данных. При перезапуске брокера Page Cache остается «горячим», обеспечивая мгновенную производительность.

Для стабильной работы под высокой нагрузкой необходим тюнинг параметров `sysctl`:

* `vm.swappiness = 1` — Минимизирует риск ухода страниц процесса в swap, что критично для производительности.
* `vm.dirty_background_ratio = 5` — Инициирует фоновый сброс (flush) грязных страниц на диск раньше, предотвращая накопление огромных объемов данных в памяти.
* `vm.dirty_ratio = 60` — Устанавливает жесткий лимит (от 60 до 80 процентов), при котором процесс блокируется до завершения синхронной записи на диск.
* `vm.max_map_count = 400000` — Kafka активно использует `mmap` для работы с индексами; стандартного лимита часто недостаточно при большом количестве партиций (рекомендуется от 400 000 до 600 000).

### Механизм Zero-Copy (`sendfile`)

Стандартная передача данных из Page Cache в сокет требует 4 переключения контекста и копирования данных в User Space. Kafka использует системный вызов `sendfile()`, реализующий Zero-Copy:

* **Standard Path:** Disk → Page Cache → User Space Buffer → Socket Buffer → NIC. (4 контекста, 4 копирования).
* **Zero-Copy Path:** Disk → Page Cache → NIC Buffer. (2 контекста, копия напрямую из Page Cache в буфер сетевой карты через дескрипторы).

## 🛡️ 3. Репликация и Консенсус (ISR модель)

Модель репликации Kafka обеспечивает баланс между консистентностью и доступностью.

* **Leader vs Follower:** В классической схеме лидер обрабатывает все запросы. Однако в современных версиях (начиная с 2.4 и выше) реализована возможность *fetching from followers*, что позволяет оптимизировать кросс-зональный (Cross-AZ) трафик и снизить расходы на передачу данных в облаках.
* **Метрики прогресса:**
* **Log End Offset (LEO):** Позиция последнего сообщения в логе конкретной реплики.
* **High Watermark (HW):** Максимальный оффсет, успешно реплицированный на все узлы из ISR.


* **Модель ISR (In-Sync Replicas):** Список реплик, которые не отстают от лидера более чем на `replica.lag.time.max.ms`. Если `acks=all`, запись считается успешной только при достижении всеми ISR-узлами, что автоматически сдвигает HW.
* **KRaft (Raft-based Consensus):** Заменяет ZooKeeper, перенося управление метаданными (Controller) внутрь самой Kafka. Это устраняет бутылочное горлышко при переизбрании лидеров и позволяет масштабировать кластер до миллионов партиций.

## 🔀 4. Гарантии доставки и Транзакции (Exactly-Once)

Для критических бизнес-процессов (FinTech) Kafka предоставляет семантику Exactly-Once (EOS).

* **Idempotent Producer:** Использование связки Producer ID (`PID`) и `Sequence Number`. Брокер дедуплицирует сообщения на уровне лога: если при сетевом сбое продюсер отправит дубликат, брокер распознает его по номеру последовательности и проигнорирует.
* **Транзакции:** Реализуются через *Transaction Coordinator* и внутренний топик `__transaction_state`.
* **Control Records:** В лог записываются специальные маркеры — `COMMIT` или `ABORT`. Если клиент работает с `isolation.level=read_committed`, он не увидит сообщения в логе, пока за ними не последует `COMMIT`-маркер.

## 🛠️ 5. Практика: Инспекция кишков Kafka через CLI

Инспекция бинарных сегментов необходима для отладки коррупции данных и анализа распределения ключей.

### Дамп сегмента лога

Команда для глубокой итерации по бинарному сегменту:

```bash
kafka-run-class.sh kafka.tools.DumpLogSegments \
  --filenames /data/kafka/logs/topic-0/00000000000000000000.log \
  --deep-iteration
```

Пример вывода (симуляция):

```text
Dumping /data/kafka/logs/topic-0/00000000000000000000.log
Starting offset: 12503
baseOffset: 12503 lastOffset: 12505 position: 0 CreateTime: 1625... size: 512 magic: 2 crc: 3245123 attributes: 0 compresscodec: NONE
| offset: 12503 CreateTime: 1625... keysize: 4 valuesize: 100 key: 001 payload: data_payload_...
| offset: 12504 CreateTime: 1625... keysize: 4 valuesize: 100 key: 002 payload: data_payload_...
```

* **Magic Byte:** Версия формата сообщения.
* **CRC:** Контрольная сумма для валидации целостности на уровне VFS/Диска.

### Дамп индекса

Проверка разреженности (sparse) индекса:

```bash
kafka-run-class.sh kafka.tools.DumpLogSegments \
  --filenames /data/kafka/logs/topic-0/00000000000000000000.index

```

Вывод покажет маппинг offset на position. Вы заметите, что записи идут не для каждого оффсета, а с шагом около 4 КБ, что подтверждает работу `log.index.interval.bytes`.

## 💰 Финтех-чеклист для собеседования

* **Вопрос:** Может ли сообщение быть потеряно, если Ack получен, но HW не сдвинулся?
* **Ответ:** Да, если `min.insync.replicas` выставлен в 1. Лидер подтвердит запись, упадет, а новый лидер (который не успел синхронизироваться) не будет иметь этой записи. Решение: `acks=all` плюс `min.insync.replicas=2`.

* **Вопрос:** Как влияет миллион уникальных ключей на Log Compaction?
* **Ответ:** Брокеры тратят значительный объем RAM на построение *Offset Map* в процессе очистки. При нехватке памяти Log Cleaner будет постоянно перезапускаться или деградировать.


* **Вопрос:** Что будет, если Consumer начнет читать старые данные, которых нет в Page Cache?
* **Ответ:** Произойдет «холодное чтение» (Disk I/O). Это вызывает резкий рост `iowait`, забивает очередь диска и негативно влияет на продюсеров, которые пишут в те же разделы بسبب конкуренции за головки HDD или IOPS SSD.

* **Вопрос:** Зачем нужен топик `__transaction_state`?
* **Ответ:** Это персистентное хранилище статусов транзакций. Если узел с *Transaction Coordinator* упадет, новый координатор восстановит состояние незавершенных транзакций из этого топика.
