#  IaC: Архитектурная битва и синергия Terraform & Ansible

> [!abstract] Стратегическая цель
> В методологии **Continuous Delivery (CD)** автоматизация — это антидот к хаосу. Цель IaC: обеспечить **прослеживаемость** (Traceability) и превратить инфраструктуру из «хрупкой снежинки» в воспроизводимый конвейер.

---

## 1. Разделение ответственности: Phoenix vs Snowflake

Для чистоты CD-конвейера необходимо разделять этапы подготовки и настройки.

###  Terraform: Provisioning (Создатель)
Его задача — **жизненный цикл ресурсов** (VM, сети, диски).
- **Концепция**: Phoenix Servers (вместо починки — уничтожение и пересоздание).
- **Результат**: Голый инстанс с сетевым доступом.

###  Ansible: Configuration Management (Настройщик)
Его задача — **вдохнуть жизнь** в ресурсы.
- **Концепция**: Snowflake Prevention (предотвращение появления уникальных, вручную настроенных серверов).
- **Результат**: Установленное ПО (JDK, Nginx), настроенные конфиги и запущенные сервисы.

## 2. Парадигмы: Декларативность vs Процедурность

| Характеристика | Terraform (Декларативный) | Ansible (Процедурный) |
| :--- | :--- | :--- |
| **Логика** | Описание «Что» (Конечный результат) | Описание «Как» (Шаги) |
| **Зависимости** | Автоматический **Граф зависимостей** | Ручной порядок выполнения задач |
| **Интеллект** | Знает состояние через **State-файл** | Проверяет систему по факту |

### HCL Пример (Terraform)
```hcl
resource "aws_instance" "app" {
  ami           = "ami-0c55b159"
  instance_type = "t2.micro"
  # Граф поймет, что сначала нужно создать SG
  vpc_security_group_ids = [aws_security_group.web.id]
}
```

### YAML Пример (Ansible)
```yaml
- name: Setup App
  tasks:
    - name: Install Java
      apt: {name: openjdk-8-jre, state: present}
    - name: Copy Artifact
      copy: {src: ./app.war, dest: /var/lib/tomcat/}
```

## 3. Управление состоянием (The State Issue)

> [!warning] Критический риск State-файла
> Потеря `terraform.tfstate` эквивалентна потере ключей от дата-центра. Инструмент «ослепнет» и не сможет управлять ресурсами.
> **Решение**: Remote Backend (S3/Consul) + State Locking.

> [!info] Идемпотентность Ansible
> Ansible не имеет файла состояния. Он каждый раз опрашивает хост. 
> *Риск*: Если выполнение прервется на середине, система может остаться в промежуточном состоянии («крах установки»).

## 4. Сравнение архитектурных моделей

| Параметр | Terraform | Ansible |
| :--- | :--- | :--- |
| **Транспорт** | Cloud API (Direct) | SSH / WinRM (Push) |
| **Агенты** | Agentless | Agentless |
| **Модель** | Immutable (Замена) | Mutable (Изменение на месте) |

---

## 5. Резюме архитектора

1.  **Terraform — это фундамент**. Используйте его для создания инфраструктуры как эфемерных сущностей.
2.  **Ansible — это настройщик**. Используйте его там, где Immutable-образы (Docker) избыточны.
3.  **Пайплайн — это закон**. Связывайте их в единый CD-конвейер. Любое ручное вмешательство — это инцидент, нарушающий прослеживаемость.
