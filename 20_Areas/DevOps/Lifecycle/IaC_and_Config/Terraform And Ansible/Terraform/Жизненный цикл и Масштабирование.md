#  Terraform: Жизненный цикл и Масштабирование (Middle-level Guide)

> [!abstract] Стратегический сдвиг
> Переход к IaC — это не просто смена CLI. Это отказ от «ручного ремесла» в пользу **воспроизводимости** и **прослеживаемости**. Инфраструктура должна быть готова к релизу в любой момент.

---

## 1. Глубокий анализ Жизненного Цикла (Lifecycle)

Понимание того, что происходит «под капотом», минимизирует риски потери **State File** — единственного источника истины.

1. **`init`**: Инициализация бэкенда и загрузка провайдеров. Обеспечивает *Confirmability*.
2. **`plan`**: Этап сопоставления. Выполняет `refresh` и вычисляет «радиус поражения» (delta).
3. **`apply`**: Реализация изменений. Здесь работает принцип **идемпотентности** — если ресурс соответствует коду, лишних вызовов API не будет.
4. **`destroy`**: Упорядоченное удаление с учетом графа зависимостей.

---

## 2. Мета-аргументы: Управление сложностью

Для Middle-специалиста критически важен выбор механизма итерации.

| Мета-аргумент | Сценарий использования | Главный риск |
| :--- | :--- | :--- |
| **`count`** | Фиксированное кол-во идентичных ресурсов. | **Index Shifting**: Удаление элемента [0] вызовет пересоздание всего списка. |
| **`for_each`** | Ресурсы на основе Map или Set. | Более сложный синтаксис обращения к атрибутам. |
| **`depends_on`** | Скрытые зависимости (когда API не видит связь). | Усложнение графа и замедление `plan`. |

> [!tip] Правило большого пальца
> Если список ресурсов может меняться в середине — **всегда** используй `for_each`. Это гарантирует стабильность состояния при рефакторинге.

## 3. Масштабирование: Модули и Изоляция

Разделение сред (dev/prod) минимизирует **Blast Radius** (радиус взрыва).

### Рекомендуемая структура проекта
```text
├── modules/
│   ├── network/       # Сетевой фундамент
│   └── compute/       # Логика вычислений
├── envs/
│   ├── dev/
│   │   ├── main.tf    # Вызов модулей для DEV
│   │   └── backend.tf # Отдельный State File (S3/Consul)
│   └── prod/
│       ├── main.tf    # Тот же код модулей, другие переменные
│       └── backend.tf # Изолированный State File (Restricted Access)
```

---

## 4. Борьба с Configuration Drift (Дрейф)

**Дрейф** — это ручные правки в консоли облака в обход кода. Это угроза стабильности системы.



### Алгоритм Remediation:
1. **Выявление**: Запуск `terraform plan` покажет расхождения.
2. **Исправление**: 
    - Либо `terraform apply` (затрет ручные правки кодом).
    - Либо `terraform import` (если правка была нужна и её надо внести в код).
3. **Предотвращение**: Доступ в консоль облака — только для чтения (Read Only).

---

## 5. Итоговый чек-лист Middle-инженера

- [ ] State-файл вынесен в Remote Backend с блокировкой (Locking).
- [ ] Все секреты (API-ключи) передаются через переменные окружения, а не зашиты в код.
- [ ] Окружения Prod и Dev физически разделены разными файлами состояния.
- [ ] Вместо `count` для разнородных ресурсов используется `for_each`.
