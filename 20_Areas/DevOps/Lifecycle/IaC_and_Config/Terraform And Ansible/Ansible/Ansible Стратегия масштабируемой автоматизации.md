#  Ansible: Стратегия масштабируемой автоматизации

> [!abstract] Философия CD
> Автоматизация инфраструктуры — это вопрос выживания бизнеса. Цель Ansible: превратить настройку серверов в **версионируемый, проверяемый и воспроизводимый** актив, устраняя «человеческий фактор» и снижая Time-to-Market.

---

## 1. Идемпотентность: Фундамент предсказуемости
В отличие от Bash-скриптов, Ansible описывает **целевое состояние**, а не процесс.

- **Математическая точность**: Результат идентичен при $N$ запусках.
- **Борьба с дрейфом (Drift)**: Если кто-то изменил конфиг вручную, Ansible вернет его к эталону.
- **Phoenix Server**: Легкость пересоздания системы с нуля вместо её «лечения».



---

## 2. Структурирование: Модульность через Роли
Следуем принципу **Convention over Configuration**. Роль — это автономный блок логики.

### Стандартная структура каталога:
- `tasks/` — Основная логика (main.yml).
- `handlers/` — Триггеры (например, restart nginx).
- `templates/` — Динамические конфиги (Jinja2).
- `defaults/` — Переменные по умолчанию (самый низкий приоритет).
- `vars/` — Константные переменные (высокий приоритет).
- `meta/` — Зависимости (например, роль `nginx` зависит от `common`).

## 3. Динамические данные и Jinja2
Использование фактов (`ansible_facts`) и шаблонов делает код адаптивным.

> [!example] Пример шаблона Nginx
> ```nginx
> server {
>     listen {{ nginx_port | default(80) }};
>     server_name {{ ansible_fqdn }}; # Автоматический сбор имени хоста
>     root /var/www/{{ project_name }};
> }
> ```

## 4. Безопасность: Ansible Vault
Хранение секретов в открытом виде — критический риск. Vault внедряет AES256 шифрование прямо в Git.

- **Traceability**: Мы видим, *кто* и *когда* изменил зашифрованный файл.
- **Команда**: `ansible-vault encrypt vars/secrets.yml`
- **Запуск**: `ansible-playbook site.yml --ask-vault-pass`



---

## 5. Гарантия качества: Тестирование с Molecule
Борьба с «регрессионными отходами» через **Shift-Left Testing**. 

1. **Spin-up**: Поднятие Docker-контейнера.
2. **Converge**: Прогон вашей роли.
3. **Idempotence**: Проверка: внесет ли второй прогон изменения? (Должно быть 0).
4. **Verify**: Тесты состояния (например, через `testinfra` или `goss`).

> [!tip] Зачем это нужно?
> Molecule дает **Rapid Feedback**. Вы узнаете об ошибке в коде за 30 секунд на локальной машине, а не через 20 минут после падения продакшена.

---

## 6. Оптимизация (Speed Tips)

> [!info] Pipelining
> Включите `pipelining = True` в `ansible.cfg`. Это заставляет Ansible выполнять задачи через одно SSH-соединение (stdin), вместо копирования файлов скриптов. Скорость возрастает в **2-4 раза**.
