# 🧪 Стратегии тестирования в CI/CD и IaC

> [!abstract] Стратегическая цель
> Превращение верификации из «узкого места» в надежный фильтр качества. Тестирование — это не этап перед релизом, а сквозной процесс контроля кода и инфраструктуры.

---

## 1. Пирамида тестирования (Test Pyramid)

Оптимизация затрат и скорости поставки требует широкого основания из быстрых тестов и минимума дорогих ручных проверок.



| Уровень теста | Инструментарий | Смысл в CD |
| :--- | :--- | :--- |
| **Unit-тесты** | JUnit, Mockito | **Regression Safety**: Мгновенная изоляция дефектов. |
| **Acceptance** | Selenium, JBehave | **Business Logic**: Проверка соответствия требованиям. |
| **Capacity** | Gatling | **Risk Mitigation**: Управление рисками нагрузки. |

> [!warning] Антипаттерн: "Рожок мороженого"
> Преобладание ручного или GUI-тестирования при малом количестве Unit-тестов. Это ведет к хрупкости пайплайна и колоссальным затратам на поддержку.

---

## 2. Тестирование Инфраструктуры (IaC Testing)

К коду инфраструктуры применяются те же стандарты, что и к прикладному ПО.

* **Molecule**: Тестирование ролей Ansible (проверка идемпотентности).
* **ChefSpec**: Unit-тестирование логики "рецептов" в симуляции.
* **ServerSpec**: Интеграционная проверка реального состояния системы (порты, пакеты) по SSH.

> [!important] Концепция "Phoenix Servers"
> Мы не латаем сервера. Мы уничтожаем их и пересоздаем из проверенного IaC-кода, исключая **Configuration Drift**.

## 3. Статический анализ и Quality Gates

**SonarQube** — "нулевой" уровень защиты. 



### Метрики качества:
1. **Цикломатическая сложность**: Чем выше сложность, тем ниже скорость поставки (Time-to-Market).
2. **Code Coverage**: Плотность "страховочной сетки".
3. **Build Breaker**: Плагин, который физически останавливает Pipeline, если код не проходит установленные пороги (Quality Gates).

---

## 4. Исследовательское тестирование (Exploratory)

Автоматизация рутины высвобождает время экспертов для интеллектуального поиска уязвимостей.

* **Миссии (Missions)**: Сфокусированные задачи по изучению функций.
* **Сессии (Sessions)**: Ограниченные по времени (time-boxed) проверки.
* **Фокус**: Юзабилити, эстетика и нестандартные сценарии отказа (Edge cases).

## 5. Сводная таблица инструментов автоматизации

| Задача | Инструмент | Уровень пайплайна |
| :--- | :--- | :--- |
| Статический анализ | **SonarQube** | Commit Stage |
| Модульные тесты | **JUnit / Mockito** | Commit Stage |
| Приемочные тесты | **HtmlUnit / BDD** | Acceptance Stage |
| Тест инфраструктуры | **Molecule / ServerSpec** | Provisioning Stage |
| Нагрузочный тест | **Gatling** | Capacity Stage |

## 📈 Итог для архитектора
Синергия автоматизации (роботы проверяют регрессию) и методологии (люди проверяют логику и удобство) создает **антихрупкую** систему, готовую к экстремальным скоростям релизов.
