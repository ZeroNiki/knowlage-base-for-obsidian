# 🏛️ Инфраструктура микросервисов: Архитектурный императив

> [!abstract] Ключевой вызов
> Переход от внутрипроцессных вызовов к сетевым требует отказа от "иллюзии контроля" в пользу динамических, самоорганизующихся и отказоустойчивых систем.

---

## 1. Проектирование API: Контрактное взаимодействие

API — это единственная точка соприкосновения сервисов. Нарушение контракта = каскадное падение системы.

### Принципы робастного дизайна (Закон Постела)
* **Будьте либеральны на приеме**: Игнорируйте неизвестные поля. Это позволяет обновлять провайдеров данных без поломки потребителей.
* **Будьте консервативны на передаче**: Строго соблюдайте опубликованную спецификацию.

> [!tip] Стандарт индустрии: Swagger (OpenAPI)
> Живая документация позволяет генерировать клиентские библиотеки и исключает расхождение между "описанием" и "реальностью".

## 2. Service Discovery: Динамическая навигация

В мире эфемерных IP-адресов (Docker/K8s) статичные конфиги — путь к катастрофе.

### Сравнительный анализ инструментов
| Инструмент | Механизм | CAP-теорема | Сценарий |
| :--- | :--- | :--- | :--- |
| **Consul** | Client/Server | **CP** (Consistency) | Service Mesh, Health Checks |
| **Eureka** | Client-side | **AP** (Availability) | Высокодинамичные облака (AWS) |
| **ZooKeeper** | Client-side | **CP** (Consistency) | Координация (Kafka/Hadoop) |

## 3. Стратегии работы с данными

Изоляция данных — критический фактор независимого деплоя.

* **Database per Component**: Каждый сервис владеет своим хранилищем. Общая БД — антипаттерн "распределенный монолит".
* **Stateless vs Stateful**: Для CD приоритетны Stateless-сервисы. Состояние выносится во внешние хранилища.
* **Schema Migrations**: Автоматизация через **Flyway/Liquibase**. Тестирование миграций на *реалистичных объемах* данных обязательно для предотвращения блокировок в продакшене.

> [!important] Концепция Phoenix Server
> Мы не обновляем сервера, мы их заменяем. Старый инстанс уничтожается, новый восстает из "золотого образа" (Immutable Infrastructure).

## 4. Паттерны отказоустойчивости: Design for Failure

Мы принимаем как аксиому: **сетевой вызов обязательно когда-нибудь упадет**.



### Защитные механизмы
1. **Circuit Breaker (Предохранитель)**: Размыкает цепь при росте ошибок, предотвращая каскадное падение и истощение ресурсов (Thread exhaustion).
2. **Retries (Повторы)**: Помогают при кратковременных сбоях. Требуют *экспоненциальной задержки*.
3. **Observability**:
    * **Метрики (Prometheus)**: Понимаем, *что* идет не так.
    * **Логи (ELK)**: Понимаем, *почему* это произошло.

---

## 📈 Итог: От MTBF к MTTR
В микросервисах мы не пытаемся сделать систему, которая никогда не падает (**Mean Time Between Failures**). Мы строим систему, которая восстанавливается мгновенно (**Mean Time To Recovery**).
