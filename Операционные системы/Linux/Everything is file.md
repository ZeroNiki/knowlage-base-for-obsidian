# 📂 Философия «Everything is a File» и архитектура ФС Linux

> [!QUOTE] Фундаментальный принцип
> С позиции системного архитектора решение «Все есть файл» обеспечивает аппаратную независимость и программный полиморфизм.

---

## 1. Унифицированная абстракция: Суть концепции
Программе не требуется знать специфику реализации драйвера: записывает ли она данные на SATA-накопитель, в сетевой сокет, в канал (pipe) или выводит их на виртуальный терминал — на уровне пользовательского пространства она взаимодействует с унифицированным слоем **VFS (Virtual File System)** через стандартные системные вызовы:
* `open()`
* `read()`
* `write()`
* `close()`

> [!SUCCESS] Профит для инженера
> Благодаря тому, что устройства ввода-вывода представлены в виде файлов, мы можем строить сложнейшие конвейеры обработки данных, используя простые атомарные инструменты и механизмы перенаправления (**redirection**). Такая архитектура радикально снижает энтропию системы.

## 2. Анатомия Inode: Индексный дескриптор как «паспорт» объекта
Файловая система на низком уровне оперирует не именами, а числовыми индексами. **Inode** — это структура метаданных, хранящая всю информацию об объекте, кроме его названия.

### Структура метаинформации inode:
* **Права доступа (Permissions):** биты чтения, записи и исполнения.
* **Идентификаторы владельца (UID/GID):** привязка объекта к субъектам системы.
* **Временные метки (Timestamps):** `atime` (доступ), `mtime` (модификация), `ctime` (изменение inode).
* **Счетчик ссылок (Link count):** количество имен (жестких ссылок), указывающих на дескриптор. 
* **Указатели на блоки данных:** адреса физических секторов на носителе.

> [!IMPORTANT] Архитектурный нюанс
> В inode **никогда не хранится имя файла**. Сама директория является специальным типом файла, содержащим таблицу соответствия: `«человекочитаемое имя» — «номер inode»`.

## 3. Механизмы связей: Hard Links и Symbolic Links

### Hard Links (Жесткие ссылки)
Создание через команду `ln`. Это дополнительное имя для уже существующего inode.
* **Механизм:** Добавление записи в таблицу директории и инкремент `link count`.
* **Ограничения:** Нельзя указывать на директории и нельзя создавать между разными ФС.

### Symbolic Links (Symlinks / Символьные ссылки)
Создаются командой `ln -s`. Это отдельный файл, содержащий текстовый путь к целевому объекту.
* **Механизм:** Система читает путь и выполняет переход. Если оригинал перемещен — ссылка «битая» (**broken**).

### Сравнительный анализ
| Параметр | Hard Link (Жесткая ссылка) | Symbolic Link (Символьная ссылка) |
| :--- | :--- | :--- |
| **Inode ID** | Совпадает с оригиналом | Различается (свой собственный) |
| **При перемещении** | Работает (связь с данными) | Обрывается (путь невалиден) |
| **Кросс-ФС** | Нет (один раздел) | Да |
| **Счетчик ссылок** | Инкрементирует у оригинала | Не влияет |

## 4. Файловая система как интерфейс
Принцип находит свое воплощение в виртуальных ФС, которые служат интерфейсом управления ядром:

* **`/proc`**: Динамическое «окно» в состояние ядра и процессов (напр. `/proc/cpuinfo`).
* **`/sys`**: Интерфейс для взаимодействия с оборудованием и драйверами.
* **Сетевая конфигурация**: Редактирование текстового файла `/etc/network/interfaces` — прямое управление через файл.
* **Абстракция DNS**: Файл `/etc/hosts` как локальная альтернатива сложным сетевым службам.

## 5. Практический инструментарий: Инструменты архитектора

> [!INFO] Инструментарий диагностики
> - **`ls -i`**: Визуализация номера inode.
> - **`stat`**: Главный инструмент инспекции. «Распаковывает» содержимое дескриптора inode.
> - **`file`**: Анализирует содержимое объекта по «магическим числам» (**magic numbers**), не полагаясь на расширение.
> - **`ln`**: Управление связями для оптимизации структуры без копирования.
