# 🧠 Git Internals: Архитектурный разбор контентно-адресуемой системы

> [!abstract] Тезис
> Git — это не просто VCS. Это **контентно-адресуемая файловая система** с интерфейсом управления версиями. Понимание «канализации» (plumbing) отделяет инженера от пользователя.

---

## 1. Философия Снимков (Snapshots)
В отличие от Delta-based систем (SVN), Git хранит **независимые снимки**.
- **Надежность**: Повреждение одного коммита не ломает всю цепочку файла.
- **Производительность**: Чтение сжатого zlib-снимка быстрее, чем реконструкция из сотен патчей.

---

## 2. Анатомия директории `.git`
Сердце репозитория и его «мозговой центр».

| Директория/Файл | Описание |
| :--- | :--- |
| `objects/` | База данных контента (Blobs, Trees, Commits). |
| `refs/` | Указатели на хеши коммитов (ветки, теги). |
| `HEAD` | Ссылка на текущую ветку (например, `ref: refs/heads/main`). |
| `index` | Бинарный файл — «черновик» следующего коммита. |
| `config` | Локальные настройки репозитория. |

---

## 3. Объектная модель: Три кита Git



### 1. Blob (Binary Large Object)
- Хранит **только контент**.
- Не знает имени файла или прав доступа.
- Дедупликация: 100 одинаковых файлов = 1 объект в базе.

### 2. Tree (Дерево)
- Аналог **директории**.
- Содержит список записей: `[права] [тип] [SHA-1] [имя]`.
- Указывает на блобы или другие деревья.

### 3. Commit (Коммит)
- Указывает на **Root Tree** (снимок проекта).
- Содержит метаданные: автор, дата, сообщение.
- Хранит SHA-1 **родителя**, формируя DAG (направленный ациклический граф).

> [!important] Принцип неизменяемости
> Объекты в Git **иммутабельны**. Любое изменение контента порождает новый SHA-1. Это гарантирует невозможность незаметного изменения истории.

---

## 4. Ссылки (Refs) и HEAD
**Ветка** — это просто текстовый файл весом 41 байт, содержащий хеш коммита.

- **Механика**: Создание ветки мгновенно, так как это просто запись файла в `refs/heads/`.
- **Detached HEAD**: Состояние, когда `HEAD` указывает напрямую на SHA-1 коммита, а не на имя ветки. 
    - *Риск:* Коммиты могут быть удалены GC (Garbage Collector), если не создать ветку.

---

## 5. Слой «Канализации» (Plumbing) vs «Фарфора» (Porcelain)

> [!info] Separation of Concerns
> Git разделяет внутренний движок и пользовательский интерфейс.

### Команды «Канализации» (для скриптов и понимания сути):
- `git hash-object`: Записать файл в базу и получить хеш.
- `git cat-file -p <hash>`: Прочитать содержимое любого объекта.
- `git ls-tree`: Посмотреть содержимое объекта-дерева.
- `git update-ref`: Ручное управление указателями веток.

---

## 6. Жизненный цикл: Три состояния
Для Middle+ критично мастерство управления **Индексом (Staging Area)**.

1. **Рабочий каталог**: Файлы в редакторе.
2. **Индекс**: Бинарный слепок для будущего коммита.
3. **База объектов**: Постоянное хранилище.

> [!tip] Partial Commits
> Используйте `git diff --staged` перед коммитом. Фиксируется состояние **Индекса**, а не файлов на диске. Это позволяет дробить одну большую правку на несколько логически чистых коммитов.
